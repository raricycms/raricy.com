# 项目架构

## 1. 项目定位

account-service 是 raricy.com 的小鱼干虚拟货币账户微服务。它从 Flask 博客单体中拆分出来，独立管理所有与"鱼干"相关的账户、余额和交易流水。

**核心职责：**

- 账户管理（开户、认领、查询）
- 复式记账转账（DEBIT / CREDIT 配对）
- 余额查询（聚合计算，非冗余存储）
- 交易流水查询（分页、筛选）

**不负责：**

- 用户身份认证（由博客系统负责）
- 业务规则判断（如签到运势值、投喂比例——由博客侧决定，account-service 只执行转账）
- 前端展示

## 4. 分层设计

### 4.1 API 层（`app/api/`）

**依赖注入（`deps.py`）：**

| 依赖 | 用途 |
|------|------|
| `get_db` | 提供异步 DB session，请求结束自动回滚异常 |
| `get_account_service` | 注入 AccountService |
| `get_transfer_service` | 注入 TransferService |
| `get_ledger_service` | 注入 LedgerService |
| `extract_api_key` | 从 `Authorization: Bearer <key>` 提取 API Key |
| `verify_internal_token` | 校验 `X-Internal-Token` 共享密钥（常数时间比较） |
| `extract_idempotency_key` | 校验 `X-Idempotency-Key` 格式（1-64 字符，仅 `[a-zA-Z0-9_-]`） |

**端点（`v1/`）：** 见 [API接口文档](API接口文档.md)。

### 4.2 服务层（`app/services/`）

| 服务 | 职责 |
|------|------|
| `AccountService` | 开户（含认领逻辑）、余额查询（单个/批量）、账户查找 |
| `TransferService` | 复式记账转账：幂等检查 → Key 验证 → 金额转换 → 行锁 → 余额校验 → 写 DEBIT/CREDIT → 保存幂等记录 |
| `LedgerService` | 分页流水查询，支持类型筛选（逗号分隔）和日期范围，含对手方和 running balance |

### 4.3 模型层（`app/models/`）

三张表，详见 [API接口文档](API接口文档.md) 中的数据模型章节。

### 4.4 核心层（`app/core/`）

| 模块 | 职责 |
|------|------|
| `currency.py` | `to_internal(external)` 乘 10000，`to_external(internal)` 除 10000，全程 Decimal |
| `security.py` | `generate_api_key()` 生成 `fish_sk_<32字节base64url>`，`hash_api_key()` SHA-256，`verify_api_key()` 常数时间比较 |
| `exceptions.py` | 6 个异常类，统一映射 HTTP 状态码 + `ErrorResponse` |
| `limiter.py` | slowapi `Limiter` 实例，`get_remote_address` 作 key function |

## 5. 复式记账原理

```
转账：系统 → 用户 A（3 鱼干）

  LedgerEntry #1: account=系统, direction=CREDIT, amount=30000
  LedgerEntry #2: account=用户A, direction=DEBIT,  amount=30000

  系统余额  = SUM(系统.DEBIT)  - SUM(系统.CREDIT)   ← 可透支
  用户A余额 = SUM(用户A.DEBIT) - SUM(用户A.CREDIT)  ← 不可透支

  结果：系统余额减少（或更负），用户A余额增加 3。
```

关键约束：
- 余额从不直接存储——永远从 `ledger_entries` 聚合计算
- `SELECT ... FOR UPDATE` 行锁防止并发双花
- 系统账户（`is_system=True`）跳过余额检查，允许透支
- 非系统账户转账前检查 `SUM(DEBIT) - SUM(CREDIT) >= transfer_amount`

## 6. 精度设计

| 概念 | 存储 | API |
|------|------|-----|
| 1 小鱼干 | BIGINT `10000` | Decimal `1.0` |
| 0.5 小鱼干 | `5000` | `0.5` |
| 0.0001 小鱼干 | `1` | `0.0001` |

转换在服务端 `currency.py` 中完成，调用方只需传递自然单位 Decimal。`amount` 字段在 API 中始终以字符串形式传输（`"3.0"`），避免浮点精度损失。

## 7. API Key 认证流程

```
1. 开户 → 生成 fish_sk_<32字节随机> → 返回明文一次
         → SHA-256(明文) 存入 accounts.api_key_hash

2. 转账 → 调用方提供 Authorization: Bearer <明文>
         → 服务端 SHA-256(明文)
         → SELECT ... WHERE api_key_hash = hash
         → 校验 account.user_id == from_user_id（防冒用）
```

特点：
- 原文不落库，只存 hash
- `secrets.compare_digest` 常数时间比较防时序攻击
- 前缀 `fish_sk_` 便于日志识别和泄露扫描
- 未认领账户 `api_key_hash = NULL`，无法作为转出方

## 8. 安全设计

| 机制 | 说明 |
|------|------|
| **X-Internal-Token** | 博客与 account-service 共享密钥，所有端点强制校验。未配置时 fail-closed（拒绝所有请求） |
| **API Key** | 每个账户独立密钥，转账时验证 `from_user_id` 归属 |
| **速率限制** | 开户 20/s、查余额 100/s、转账 10/s、查流水 30/s |
| **行锁** | `SELECT ... FOR UPDATE` 防并发超扣 |
| **幂等** | `X-Idempotency-Key` 24h TTL，相同 key 返回缓存结果，不同 body 返回 409 |
| **请求追踪** | 每个请求分配 `X-Request-ID`，错误响应包含 `request_id` |

## 9. 与博客系统的关系

```
博客 (Flask)                          account-service (FastAPI)
───────────                          ────────────────────────
用户注册 ──POST /accounts──────────→ 创建账户，返回 API Key
用户签到 ──POST /transfers─────────→ 系统账户 → 用户（checkin）
用户投喂 ──POST /transfers─────────→ 投喂者 → 被投喂者（feed_out / feed_income）
管理员发放──POST /transfers────────→ 系统账户 → 用户（admin_grant）
显示余额 ──GET /balance────────────→ 聚合查询
显示流水 ──GET /ledger─────────────→ 分页查询
```

博客系统在账本中就是 `raricy-blog-system` 账户。它不是外部"神"操纵余额，而是账本的平等参与者——唯一特殊之处是 `is_system=True` 允许透支（发行新鱼干）。
