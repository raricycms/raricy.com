# 01 — 背景与现状分析

## 1. 当前实现回顾

### 1.1 余额存储：挂在 User 表上

```python
# app/models/user.py:44
dried_fish = db.Column(db.Integer, default=0, nullable=False)
# 高精度整型，1 小鱼干 = 10000 单位
```

用户余额是 `users` 表的一个普通字段。查询余额就是查用户，修改余额就是 UPDATE users。

### 1.2 交易流水：单边记录

```python
# app/models/fish.py — FishTransaction 表
class FishTransaction(db.Model):
    __tablename__ = 'fish_transactions'
    id = db.Column(db.Integer, primary_key=True)
    user_id = db.Column(db.String(36), db.ForeignKey('users.id'), ...)
    amount = db.Column(db.Integer, nullable=False)  # 正=获得, 负=消费
    type = db.Column(db.String(32), ...)   # checkin / admin_grant / purchase / ...
    description = db.Column(db.String(255), ...)
    # ...
```

每条流水只记录**一个用户**的变动，没有对手方、没有借贷方向。这是"单式记账"——够用但不可审计。

### 1.3 服务层

```python
# app/service/fish.py
add_fish(user_id, amount, type, ...)     # 原子 UPDATE + 写流水
deduct_fish(user_id, amount, type, ...)  # 原子 UPDATE + 写流水（WHERE balance >= amount 防超扣）
get_balance(user_id)                     # 直接读 User.dried_fish
get_transactions(user_id, page, ...)     # 分页查流水
get_balance_leaderboard(limit)           # 排行榜
```

服务层已经做了一些正确的事：**原子 UPDATE 防超扣**、**整型存储避免浮点精度问题**。但它仍然操作的是博客数据库里的表。

### 1.4 消费方（谁在用鱼干）

| 模块 | 文件 | 操作 |
|------|------|------|
| 签到 | `app/service/checkin.py:170` | `add_fish(user_id, fortune_val * 10000, 'checkin', ...)` |
| 文章投喂 | `app/web/blog/services/feed_fish_service.py:47-59` | `deduct_fish(from_user)` → `add_fish(to_author, 20%)` |
| 管理后台 | CLI 命令 `flask fish grant/deduct` | 直接调 `add_fish` / `deduct_fish` |
| 前端展示 | `app/web/auth/fish.py` | 读余额、读流水、排行榜 |

### 1.5 投喂记录：额外的一张表

```python
# app/models/blog_feed.py — BlogFeed
class BlogFeed(db.Model):
    __tablename__ = 'blog_feeds'
    blog_id + user_id 唯一约束
    amount  # 累计投喂条数，单篇上限 5
```

投喂本身是博客域的概念（"给文章投小鱼干"），但金额变动在鱼干系统里。两者通过 `feed_fish_service.py` 手动协调，不是事务性的跨域操作（虽然在同一 DB 里用 `auto_commit=False` 勉强做到了原子性）。

## 2. 当前架构的 7 个问题

### 问题 1：余额与用户身份耦合

User 表承载了太多职责：认证、角色、禁言、通知偏好、运势统计、**还有余额**。这意味着：

- 任何查 User 的查询都可能无意中带上 `dried_fish`
- 想给鱼干单独做缓存策略？做不到——它和用户身份是一张表
- 想给鱼干做读写分离？也做不到
- 未来如果要支持"一个用户多个币种"，需要改 User 表的 schema

### 问题 2：单库单连接池，资源竞争

博客内容（文章、评论、通知）和虚拟货币交易共享同一个 SQLite/PostgreSQL 实例。将来模拟炒股上线后——订单簿、持仓、撮合日志——全部涌入同一个 DB。高频交易操作会和博客的内容查询竞争连接池和 IO。

### 问题 3：非复式记账，余额为可变字段

```
当前：User.dried_fish += amount   # 直接改余额
应该：INSERT INTO ledger_entries (debit) + INSERT INTO ledger_entries (credit)
       balance = SUM(ledger_entries WHERE account_id = X)
```

可变字段的问题：
- **无法审计**：你怎么知道某用户 3 个月前的余额是怎么来的？只能人工翻流水
- **无法回溯**：不能查询"截至 2026-06-01 的余额"
- **篡改风险**：直接 UPDATE 一行就能改余额，没有复式校验
- 好消息是当前代码用 `FishTransaction` 记录了流水，但这只是日志，不是余额的来源

### 问题 4：无幂等键

```python
# 签到：如果网络超时，用户点了两次
add_fish(user_id, fortune_val * 10000, 'checkin', ...)
# → 可能加两次！
```

当前通过签到日期 (`checkin_date` 唯一) 和投喂的 `auto_commit=False` 事务来缓解，但这不是系统性的解决方案。一个正经的支付系统，每个请求都应该带 `idempotency_key`。

### 问题 5：精度单位泄漏

```python
# 每个消费者都要做转换：
'balance': current_user.dried_fish / FISH_UNIT       # auth/fish.py:75
'dried_fish': result.get('dried_fish', 0) / FISH_UNIT  # checkin/__init__.py:92
tx['amount'] = tx['amount'] / FISH_UNIT                # auth/fish.py:90
```

`FISH_UNIT = 10000` 是内部实现细节。但它在至少 5 个文件中被 import 和使用。如果以后想改成 `FISH_UNIT = 100`（降低精度需求），要改所有调用方。正确的做法是服务端只接受和返回"1 小鱼干 = 1 单位"，内部精度对调用方透明。

### 问题 6：公开 API 无鉴权

```python
@auth_bp.route('/fish/api/balance/<user_id>')  # 无需登录
def api_user_balance(user_id):
    balance = get_balance(user_id)  # 任意用户的余额
    return jsonify({'balance': balance / FISH_UNIT})

@auth_bp.route('/fish/api/balance/batch', methods=['POST'])  # 也无需登录
def api_batch_balance():
    ...
```

这些"公开接口"的设计初衷可能是方便前端调用。但没有任何认证意味着任何人都能查询任意用户的余额。对于虚拟货币系统，这至少需要服务间 API Key 验证。

### 问题 7：扩展"模拟炒股"时无处安放

想象在现有架构上加模拟炒股，你需要：

- `StockPosition` 表 → 放博客 DB？
- `Order` 表（买单/卖单）→ 放博客 DB？
- 撮合引擎 → 在 Flask 进程里跑 Celery？
- 行情数据缓存 → 放博客 Redis？

这些全都**不属于博客域**。把它们塞进博客项目会让两个系统的边界越来越模糊，最终谁都维护不了。

## 3. 重构目标

| 目标 | 具体含义 |
|------|----------|
| **独立部署** | 账户服务有自己的进程、数据库、连接池 |
| **复式记账** | 余额从 `LedgerEntry` 聚合，不存储可变余额字段 |
| **幂等安全** | 所有写操作支持 `X-Idempotency-Key` |
| **服务间鉴权** | 账户服务不暴露公网端口。双层认证：X-Internal-Token（所有端点）+ API Key（转账端点）。实际实现中新增了 X-Internal-Token 共享密钥层，防止端口意外暴露 |
| **精度透明** | 调用方使用自然单位（1 小鱼干 = 1.0），内部精度由服务端管理 |
| **扩展友好** | 多币种、模拟炒股、打赏等新功能在账户服务内部添加，不污染博客代码 |

## 5. 实际实现补充说明

> 账户服务的实际实现（2026-07）在以下方面超出了本文档的设计范围：

| 方面 | 设计计划 | 实际实现 |
|------|----------|----------|
| 安全层 | 仅 API Key | **新增** X-Internal-Token 双层认证 |
| 数据库 | PostgreSQL only | SQLite 默认，PostgreSQL 可选 |
| API 范围 | Phase 1 不实现流水查询/批量查询 | Phase 1 即实现了全部 6 个端点 |
| 未认领账户 | 未涉及 | 转账接收方自动创建未认领账户，用户通过 `POST /accounts` 认领 |

以上补充不影响本文档对现状问题和重构目标的诊断（这些问题仍然是真实的），仅反映实施过程中的工程决策。
