# 02 — 目标架构设计

## 1. 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                        公网                                 │
│                          │                                  │
│                    Nginx (:443)                             │
│                          │                                  │
│              ┌───────────┴───────────┐                      │
│              │                       │                      │
│         /, /auth, /blog, ...    (不暴露公网)                 │
│              │                       │                      │
│     ┌────────▼────────┐    ┌────────▼─────────┐             │
│     │  raricy.com     │    │  account-service │             │
│     │  (Flask :5000)  │───▶│  (FastAPI :8000) │             │
│     │                 │    │                   │             │
│     │  - 用户认证      │    │  - 账户管理        │             │
│     │  - 博客 CRUD    │    │  - 复式记账流水     │             │
│     │  - 签到（触发）  │    │  - 转账/扣款       │             │
│     │  - 投喂（触发）  │    │  - 余额查询        │             │
│     │  - 页面渲染      │    │  - 幂等控制        │             │
│     └────┬────────────┘    └────────┬──────────┘             │
│          │                          │                        │
│     ┌────▼────────┐          ┌──────▼──────────┐             │
│     │  PostgreSQL  │          │  SQLite (默认)  │             │
│     │  (博客 DB)   │          │  / PostgreSQL   │             │
│     └─────────────┘          │  (可选)         │             │
│                              └─────────────────┘             │
│                                                             │
│   Docker Compose 管理所有服务                                 │
└─────────────────────────────────────────────────────────────┘
```

核心原则：**博客是"触发方"，账户服务是"记账方"**。

- 博客说："用户 A 签到获得了 3 个小鱼干"
- 账户服务执行：系统账户 → 用户 A 账户，转账 3 小鱼干，记录流水
- 博客不需要知道小鱼干怎么存的、精度多少、流水表长什么样

## 2. 服务边界

```
┌─────────────────────────────────┬─────────────────────────────────┐
│         博客 (Flask)             │      账户服务 (FastAPI)           │
├─────────────────────────────────┼─────────────────────────────────┤
│ ✅ 用户注册 / 登录 / 认证        │ ✅ 账户创建（用户注册时调用）      │
│ ✅ 用户身份管理（角色/禁言等）    │ ✅ 余额管理（从流水聚合）          │
│ ✅ 签到逻辑（何时签、运势值多少）│ ✅ 转账（系统→用户、用户→用户）    │
│ ✅ 投喂逻辑（谁投谁、上限多少）  │ ✅ 交易流水记录（复式记账）        │
│ ✅ 文章管理                      │ ✅ 流水查询 / 分页               │
│ ✅ 页面渲染（余额展示等）        │ ✅ 幂等控制                       │
│ ✅ 触发资金变动                  │ ✅ 排行榜（余额维度）             │
│ ❌ 不再直接读写鱼干余额           │ ❌ 不关心用户密码、角色           │
│ ❌ 不再写鱼干交易流水             │ ❌ 不关心文章、投喂业务含义        │
└─────────────────────────────────┴─────────────────────────────────┘
```

### 博客调用账户服务的场景

| 场景 | 触发点 | 调用 API | 认证方式 |
|------|--------|----------|----------|
| 用户注册 | `auth/views.py` 注册成功后 | `POST /accounts` 创建账户（返回 API Key） | X-Internal-Token |
| 每日签到 | `checkin.py` 签到领运势 | `POST /transfers`（博客系统用自己的 API Key 发鱼干） | X-Internal-Token + 博客系统 API Key |
| 文章投喂 | `feed_fish_service.py` | `POST /transfers`（博客代用户用其 API Key 转账） | X-Internal-Token + 用户 API Key |
| 管理后台 | CLI 命令 `flask fish grant` | `POST /transfers`（博客系统 API Key） | X-Internal-Token + 博客系统 API Key |
| 余额展示 | 各页面的余额/排行榜 | `GET /accounts/{user_id}/balance` | X-Internal-Token |
| 流水分页 | `/auth/fish/transactions` | `GET /accounts/{user_id}/ledger` | X-Internal-Token |

> **关键区别**：签到用的是**博客系统自己的 Key**（因为出钱的是博客系统），投喂用的是**用户的 Key**（因为出钱的是用户）。所有请求还必须携带 `X-Internal-Token` 共享密钥——这是服务间的"门禁卡"，即使端口被意外暴露，没有此密钥的外部请求也会被直接拒绝。

## 3. 通信协议

### 3.1 传输层

- **协议**：HTTP/1.1 + JSON
- **内网通信**：Docker 内部网络，不经过 Nginx
- **超时**：博客侧设置 5s 超时（转账是同步的，不应长时间阻塞）

### 3.2 认证：统一的 API Key 模型

**核心理念：没有特权方。一切都是账户，一切认证都用 API Key。**

```
┌──────────────────────────────────────────────────────────┐
│                      账户服务                             │
│                                                          │
│   Account: "raricy-blog-system"                          │
│   API Key:  "fish_sk_xxxxxxxxxxxx"                       │
│   余额:     9999999.0 小鱼干（取之不尽）                    │
│                                                          │
│   Account: "user-a"                                      │
│   API Key:  "fish_sk_yyyyyyyyyyyy"                       │
│   余额:     15.5 小鱼干                                   │
│                                                          │
│   Account: "user-b"                                      │
│   API Key:  "fish_sk_zzzzzzzzzzzz"                       │
│   余额:     3.0 小鱼干                                    │
│                                                          │
│   ── 认证规则（唯一）：                                    │
│   转账时，请求头必须携带 from_user 的 API Key              │
│   Authorization: Bearer fish_sk_...                      │
│   账户服务验证：Key 存在 → Key 所属账户 = from_user_id       │
│   ✓ 通过 → 执行转账                                       │
└──────────────────────────────────────────────────────────┘
```

**博客系统也是一个普通账户**，只是余额极大（或在账户服务中标记为可无限透支）。签到发鱼干时：

```
博客用自己的 API Key 调转账 API：
  from_user_id: "raricy-blog-system"
  to_user_id:   "user-a"
  amount:       3.0

账户服务验证："raricy-blog-system" 的 API Key ✓ → 扣博客系统的钱 → 加给用户 A
```

投喂时：

```
博客用用户 A 的 API Key 调转账 API（博客替用户 A 保管 Key）：
  from_user_id: "user-a"
  to_user_id:   "user-b" (作者)
  amount:       5.0

账户服务验证："user-a" 的 API Key ✓ → 扣用户 A 的钱 → 加给作者 B
```

**这意味着：**
- 没有 HMAC、没有特殊签名——只有一种认证方式
- 未来任何应用（手机 App、第三方插件），只要能提供用户的 API Key，就能替用户发起交易
- 博客被攻破 → 攻击者拿到博客存储的用户 Key → 可以转走用户的鱼干，但账户服务本身没有后门
- 用户也可以自己保管 Key（不用博客托管），完全脱离博客操作账户

**API Key 格式：**

```
fish_sk_<32字节随机数 base64url 编码>
例：fish_sk_wPX2qV8zN7rKdLmY3xBhFjRtWcAeUoSn
```

前缀 `fish_sk_` 便于日志扫描和泄露检测（类似 GitHub 的 `ghp_` 前缀）。

> **实际实现补充**：在上述 API Key 认证之上，实际实现新增了 **X-Internal-Token** 层（详见 [3.5 节](#35-x-internal-token-安全层新增)）。所有端点（除 `/health`）还需要携带博客与账户服务之间的共享密钥。这是实施过程中的安全增强。

### 3.3 幂等：X-Idempotency-Key

所有 `POST /transfers` 请求必须携带：

```
X-Idempotency-Key: <uuid-v4>
```

账户服务行为：
1. 收到请求，先查 `idempotency_keys` 表
2. 如果 key 存在 → 直接返回之前的响应（幂等）
3. 如果 key 不存在 → 执行转账，写入 key + 响应，返回结果

幂等键在服务端保留 24 小时后自动清理（TTL）。

### 3.4 响应格式

```json
// 成功
{
    "code": 200,
    "data": {
        "transaction_id": "uuid",
        "from_balance": "95.0",
        "to_balance": "105.0"
    },
    "request_id": "uuid",
    "message": "ok"
}

// 客户端错误
{
    "code": 400,
    "message": "小鱼干不足",
    "detail": {
        "user_id": "xxx",
        "required": 5.0,
        "available": 3.0
    },
    "request_id": "uuid"
}

// 服务端错误
{
    "code": 500,
    "message": "内部错误",
    "request_id": "uuid"
}
```

### 3.5 X-Internal-Token 安全层（新增）

在实际实现中，所有 API 端点（除 `/health` 外）额外要求 `X-Internal-Token` 请求头。这是博客与账户服务之间的**共享密钥**，提供额外一层网络隔离安全。

**为什么需要？**

原计划仅依赖 API Key 认证和 Docker 网络隔离（端口绑定 `127.0.0.1`）。但实际操作中：
- 端口可能被意外暴露（配置错误、Nginx 反代规则遗漏）
- Docker 网络可能被意外桥接到宿主机
- 增加一层共享密钥是零成本的防御纵深

**实现方式：**

```
X-Internal-Token: <shared-secret>
```

- 由 `INTERNAL_TOKEN` 环境变量配置（账户服务侧）
- 博客侧由 `ACCOUNT_SERVICE_INTERNAL_TOKEN` 配置
- 使用 `secrets.compare_digest` 常数时间比较，防止时序攻击
- **Fail-closed**：如果环境变量未配置，服务拒绝所有请求（不 fallback 到开放模式）

**认证层次：**

```
X-Internal-Token （验证调用方是可信的后端服务）
       +
API Key / Bearer Token （验证 from_user 身份 — 仅转账端点）
```

两层认证各司其职：X-Internal-Token 守"门"（谁可以调这个服务），API Key 守"钱"（谁的钱可以被花）。

## 4. 部署拓扑

### 4.1 Docker Compose（开发/生产统一）

**默认配置（SQLite — 无需独立数据库容器）：**

```yaml
# docker-compose.yml（新增部分）
services:
  # 已有
  blog:
    build: .
    ports:
      - "5000:5000"
    environment:
      - ACCOUNT_SERVICE_URL=http://account:8000
      - ACCOUNT_SYSTEM_KEY=${ACCOUNT_SYSTEM_KEY}
      - ACCOUNT_SERVICE_INTERNAL_TOKEN=${INTERNAL_TOKEN}
    depends_on:
      - blog-db
      - account

  # 新增
  account:
    build: ./account-service
    ports:
      - "127.0.0.1:8000:8000"   # 仅本地回环，不暴露公网
    environment:
      - DATABASE_URL=sqlite+aiosqlite:///data/account.db
      - INTERNAL_TOKEN=${INTERNAL_TOKEN}
    volumes:
      - account_data:/app/data    # 持久化 SQLite 文件

volumes:
  account_data:
```

**可选：PostgreSQL 替代方案**

如需更高并发或跨实例部署，可将 `DATABASE_URL` 切换为 PostgreSQL：

```yaml
  account:
    environment:
      - DATABASE_URL=postgresql+asyncpg://user:pass@account-db:5432/account
      - INTERNAL_TOKEN=${INTERNAL_TOKEN}
    depends_on:
      - account-db

  account-db:
    image: postgres:16-alpine
    volumes:
      - account_db_data:/var/lib/postgresql/data
    environment:
      - POSTGRES_DB=account
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=${ACCOUNT_DB_PASSWORD}
```

### 4.2 网络隔离

- `account` 服务的 8000 端口只绑定 `127.0.0.1`，公网无法直接访问
- 博客通过 Docker 内部 DNS（`http://account:8000`）调用账户服务
- 所有请求需要 `X-Internal-Token` 共享密钥作为额外安全层

## 5. 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 同步还是异步通信？ | **同步 HTTP** | 转账必须确认结果才能响应用户；如果用消息队列，用户点了"投喂"后不知道是否成功 |
| 余额存不存？ | **不存，从流水聚合** | 复式记账的核心原则。平衡性通过流水 SUM 验证 |
| 账户服务的 user_id 是什么？ | **博客的 user.id (UUID)** | 账户服务不自己生成用户 ID，直接用博客的。这是"外部 ID"模式，账户服务不拥有用户身份 |
| 一个用户能有多个账户吗？ | **一个用户 + 一个币种 = 一个账户** | `(user_id, currency)` 唯一。当前只有 `DRIED_FISH` 币种，未来可扩展 |
| 事务跨服务怎么办？ | **博客侧先准备好业务数据，再调账户服务** | 例如投喂：先验证文章存在、投喂未超限（博客侧），再调转账 API（账户侧）。如果转账失败，博客侧业务数据不回滚（最终一致性） |
| 服务间如何认证？ | **双层：X-Internal-Token + API Key** | X-Internal-Token 验证调用方是可信后端服务，API Key 验证 from_user 身份。两层各司其职，纵深防御 |
