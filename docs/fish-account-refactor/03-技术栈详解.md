# 03 — 技术栈详解

## 1. 选型对比

在选择账户服务的技术栈时，考察了 5 个方案：

| 方案 | 优势 | 劣势 | 适合度 |
|------|------|------|--------|
| **FastAPI (Python)** | 与博客技术栈一致；Pydantic 类型安全；自动 OpenAPI 文档；原生 async | Python GIL，CPU 密集场景受限 | ⭐⭐⭐⭐⭐ |
| Go (Gin/Fiber) | 极高性能；编译单文件部署；goroutine 并发模型 | 学习曲线陡峭；生态不如 Python 丰富；ORM 不够成熟 | ⭐⭐⭐ |
| Node.js (Express/Fastify) | 异步原生；NPM 生态大；JSON 处理天然 | 回调/Promise 链历史包袱；TypeScript 配置繁琐；数字精度需额外处理 | ⭐⭐ |
| 继续用 Flask | 零学习成本；代码直接复用 | 无原生 async；无自动 API 文档；手动参数校验；本质上没拆分 | ⭐ |
| Rust (Axum) | 极致性能；零成本抽象；内存安全 | 开发速度慢；人才稀缺；杀鸡用牛刀 | ⭐ |

**结论：FastAPI 是最佳选择。**

核心理由不是"Python 简单"，而是它恰好解决了当前 Flask 架构的痛点，同时在你的规模下性能绰绰有余。

## 2. FastAPI 如何解决 Flask 痛点

### 痛点 1 → Pydantic v2 类型安全

```python
# Flask 当前写法（手动校验，手动序列化）
page = request.args.get('page', 1, type=int)
per_page = request.args.get('per_page', 20, type=int)
balance = current_user.dried_fish / FISH_UNIT  # 手动转换

# FastAPI 写法（自动校验 + 自动序列化）
@router.get("/accounts/{user_id}/balance", response_model=BalanceResponse)
async def get_balance(
    user_id: str,
    currency: str = Query("DRIED_FISH"),
    service: AccountService = Depends(get_account_service),
):
    balance = await service.get_balance(user_id, currency)
    return BalanceResponse(user_id=user_id, balance=balance, currency=currency)
```

- `user_id` 自动从路径提取
- `currency` 自动从 query string 提取并校验
- `response_model` 自动把返回值序列化为 JSON，`Decimal` → `float` 转换在 schema 层完成
- 调用方永远拿到 1 小鱼干 = 1.0，内部精度完全透明

### 痛点 2 → 自动 OpenAPI / Swagger 文档

FastAPI 从代码自动生成 OpenAPI 3.1 schema。启动后访问 `http://account-service:8000/docs` 就能看到交互式 API 文档。

这对于**博客和账户服务之间的 API 契约**极其重要。博客开发者不需要翻账户服务的代码，只要看 Swagger 就能知道有哪些接口、参数是什么、返回什么。

### 痛点 3 → 原生 async/await

```python
# 转账操作涉及多次 DB 查询，async 让 IO 等待不阻塞其他请求
async def transfer(self, from_user_id, to_user_id, amount, idempotency_key):
    async with self.db.begin() as tx:
        # 1. 检查幂等
        existing = await self._check_idempotency(idempotency_key)  # await DB
        # 2. SELECT FOR UPDATE 锁两行
        from_acc = await self._lock_account(from_user_id)           # await DB
        to_acc = await self._lock_account(to_user_id)               # await DB
        # 3. 写入两条 ledger entry
        await self._insert_entries(tx_id, from_acc, to_acc, amount) # await DB
        # 4. 记录幂等键
        await self._save_idempotency(idempotency_key, tx_id)       # await DB
```

在 Flask 中这些操作会阻塞当前 worker。FastAPI + asyncpg 让一个 worker 可以交错处理多个请求的 IO 等待。

### 痛点 4 → 依赖注入

```python
# app/api/deps.py
async def get_db() -> AsyncGenerator[AsyncSession, None]:
    async with async_session() as session:
        yield session

async def get_account_service(
    db: An AsyncSession = Depends(get_db)
) -> AccountService:
    return AccountService(db)

# 路由中使用
@router.post("/transfers")
async def transfer(
    request: TransferRequest,
    service: AccountService = Depends(get_account_service),
):
    return await service.transfer(request)
```

对比 Flask 当前的做法——在路由函数内部 `from app.service.fish import add_fish` 然后直接调——FastAPI 的依赖注入让代码更可测试（test 时可以 mock `get_db`）。


## 5. 性能评估

### 5.1 为什么 asyncpg 比同步驱动快

同步模式（psycopg2）：

```
Worker 1: [发送查询] → [等待 DB 响应] → [收到结果] → [处理下一个请求]
               ↑ 这段时间 worker 什么都不做
```

异步模式（asyncpg）：

```
Worker 1: [发查询A] → [发查询B] → [收结果A] → [发查询C] → [收结果B] → ...
               ↑ DB 处理 A 时，worker 可以发 B
```

对于账户服务——大部分请求都是 DB 查询（查余额、查流水、写交易）——async 能在相同 worker 数下提升 2-5 倍吞吐。

### 5.2 预估性能

参考数据（FastAPI + asyncpg + PostgreSQL，单实例，2 vCPU）：

| 操作 | 预估 QPS |
|------|----------|
| `GET /balance/{user_id}` (单行聚合) | ~3000 QPS |
| `POST /transfers` (事务 + 2 INSERT) | ~800 QPS |
| `GET /ledger/{user_id}` (分页查询) | ~1500 QPS |

这些是保守估计。你的博客有多少并发用户？几十个。账户服务轻松应对，即使未来模拟炒股上线，每秒几百笔撮合也不在话下。

### 5.3 什么时候需要考虑换 Go？

- 单实例 QPS 超过 5000（你可能永远到不了）
- 需要极低的 p99 延迟（<1ms，Python 做不到）
- 需要 WebSocket 长连接管理数万并发（Python 这方面确实弱）

这些都不适用于你的场景。FastAPI 是正确选择。

## 6. 数据库选择

### PostgreSQL（唯一选择）

**不用 SQLite 的理由：**
- 账户服务有并发写入需求（两人同时转账不能冲突），SQLite 的写锁是库级
- `SELECT ... FOR UPDATE` 行级锁是复式记账原子操作的基石，SQLite 支持有限
- asyncpg 只支持 PostgreSQL

**版本**：PostgreSQL 16，利用其性能改进（更快的聚合查询对 `SUM(ledger_entries)` 有帮助）。

### 连接池

```python
# 使用 SQLAlchemy 2.0 async engine + asyncpg 连接池
engine = create_async_engine(
    DATABASE_URL,
    pool_size=10,         # 常驻连接数
    max_overflow=20,      # 突发额外连接
    pool_pre_ping=True,   # 连接健康检查
)
```

10+20 的连接池对于你预估的负载足够了。
