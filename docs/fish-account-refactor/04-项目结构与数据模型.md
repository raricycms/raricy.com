## 2. 数据模型设计（核心）

### 2.1 设计原则

1. **余额不存储** — `Account` 表没有 `balance` 列。余额 = `SUM(LedgerEntry)` 的聚合结果
2. **复式记账** — 每笔转账产生最少 2 条 `LedgerEntry`：一条 DEBIT（出账），一条 CREDIT（入账）
3. **交易不可变** — `LedgerEntry` 写入后永远不 UPDATE/DELETE。冲正通过创建新的反向交易实现
4. **幂等键** — `idempotency_keys` 表存储已处理的请求，防止重复执行
5. **API Key 认证** — 每个账户有一个 API Key（`fish_sk_` 前缀）。转账时验证 `Bearer token` 属于 `from_user_id`

### 2.2 Account 表

```sql
-- 账户表
CREATE TABLE accounts (
    id          UUID PRIMARY KEY,              -- 默认 gen_random_uuid() 或应用层生成
    user_id     VARCHAR(36)  NOT NULL,          -- 外部系统的用户 ID（如博客的 user.id）
    currency    VARCHAR(20)  NOT NULL DEFAULT 'DRIED_FISH',
    api_key_hash VARCHAR(128),                 -- API Key 的 SHA-256 哈希。NULL = 未认领账户
    api_key_prefix VARCHAR(12),                 -- API Key 前 12 字符。NULL = 未认领账户
    is_system   BOOLEAN NOT NULL DEFAULT FALSE, -- 是否为系统账户（可透支）
    created_at  DATETIME  NOT NULL DEFAULT (datetime('now')),

    -- 一个用户在同一个币种下只有一个账户
    CONSTRAINT uq_user_currency UNIQUE (user_id, currency)
);

CREATE INDEX idx_accounts_user_id ON accounts(user_id);
CREATE INDEX idx_accounts_api_key_hash ON accounts(api_key_hash);
```

> **实际实现说明**：`api_key_hash` 和 `api_key_prefix` 为 NULLABLE。当转账接收方没有账户时，系统自动创建"未认领"账户（`api_key_hash = NULL`），该账户可以接收资金但不能发起转账。用户通过 `POST /accounts` 认领时生成 API Key 并填写这两个字段。

**注意：没有 `balance` 列。**

为什么不存余额？

| 存余额 (mutable) | 不存余额 (derived) |
|---|---|
| `UPDATE accounts SET balance = ?` | `SELECT SUM(...) FROM ledger_entries` |
| 修改历史不可知 | 任意时间点可回溯 |
| 和流水可能不一致（bug/并发） | 流水是唯一真相源 |
| 需要额外的对账逻辑 | 天然一致 |

### 2.3 LedgerEntry 表（核心）

```sql
-- 复式记账流水表
CREATE TABLE ledger_entries (
    id              UUID PRIMARY KEY,            -- 应用层生成 UUID
    transaction_id  UUID NOT NULL,              -- 同一笔交易的多条 entry 共享此 ID
    account_id      UUID NOT NULL REFERENCES accounts(id) ON DELETE RESTRICT,
    direction       VARCHAR(6) NOT NULL CHECK (direction IN ('DEBIT', 'CREDIT')),
    amount          BIGINT NOT NULL,            -- 高精度整型（1 小鱼干 = 10000 单位）
    currency        VARCHAR(20) NOT NULL DEFAULT 'DRIED_FISH',
    entry_type      VARCHAR(32) NOT NULL,       -- 业务类型（不是方向）
    description     VARCHAR(255),
    metadata        JSON DEFAULT '{}',          -- 扩展元数据（数据库列名 `metadata`；Python 属性 `metadata_`，因 `metadata` 是 SQLAlchemy 保留字）
    created_at      DATETIME NOT NULL DEFAULT (datetime('now')),

    CONSTRAINT chk_amount_positive CHECK (amount > 0)
);

-- 查询索引
CREATE INDEX idx_ledger_account_created ON ledger_entries(account_id, created_at DESC);
CREATE INDEX idx_ledger_tx_id ON ledger_entries(transaction_id);
CREATE INDEX idx_ledger_type_created ON ledger_entries(entry_type, created_at DESC);

-- 幂等键表
CREATE TABLE idempotency_keys (
    key             VARCHAR(64) PRIMARY KEY,     -- 外部传入的幂等键（1-64 字符）
    transaction_id  UUID,                        -- 对应的交易 ID（成功时写入）
    response_json   JSON,                        -- 首次请求的响应（幂等返回时使用）
    created_at      DATETIME NOT NULL DEFAULT (datetime('now')),
    expires_at      DATETIME NOT NULL            -- TTL（默认 24h），到期后自动清理
);

CREATE INDEX idx_idempotency_expires ON idempotency_keys(expires_at);
```

### 2.4 API Key 设计

**为什么是 API Key 而不是 HMAC？**

之前的设计中，博客和账户服务共享一个 HMAC Secret，账户服务无条件信任博客。这意味着账户服务不是真正独立的——它只是博客的附属记账模块。

统一使用 API Key 之后：**每个账户（包括博客系统本身）都有自己的 Key。转账时，必须提供 `from_user` 的 Key 才能从该账户扣款。** 规则一致，没有特权通道。

```python
# core/security.py — API Key 生成与验证

import secrets
import hashlib

def generate_api_key() -> tuple[str, str, str]:
    """生成 API Key。返回 (明文 Key, 哈希值, 前缀)。"""
    raw = secrets.token_urlsafe(32)
    key = f"fish_sk_{raw}"
    key_hash = hashlib.sha256(key.encode()).hexdigest()
    prefix = key[:12]  # "fish_sk_wPX2..."
    return key, key_hash, prefix

def hash_api_key(key: str) -> str:
    """对 API Key 做 SHA-256 哈希（存储用）。"""
    return hashlib.sha256(key.encode()).hexdigest()

def verify_api_key(plain_key: str, stored_hash: str) -> bool:
    """验证 API Key。常数时间比较。"""
    return secrets.compare_digest(
        hashlib.sha256(plain_key.encode()).hexdigest(),
        stored_hash,
    )
```

**关键安全属性：**

| 属性 | 实现 |
|------|------|
| 数据库不存明文 Key | 只存 `SHA-256(key)` 的哈希 |
| 无法从数据库反推 Key | SHA-256 单向，数据库泄露也拿不到 Key |
| Key 可识别 | `fish_sk_` 前缀，方便日志扫描和 GitHub 泄露检测 |
| 常数时间比较 | `secrets.compare_digest` 防止时序攻击 |
| Key 归属绑定 | 验证时间时检查 `Account.api_key_hash` 且 `Account.user_id == from_user_id` |

### 2.5 理解复式记账：一笔转账的记录

假设系统给用户 A 发 3 个小鱼干作为签到奖励：

```
LedgerEntry 1:
  transaction_id = "tx-001"
  account_id     = SYSTEM_ACCOUNT
  direction      = "CREDIT"           ← 系统支出（钱离开系统）
  amount         = 30000  (3 × 10000)
  entry_type     = "checkin"
  description    = "系统签到支出"

LedgerEntry 2:
  transaction_id = "tx-001"
  account_id     = USER_A_ACCOUNT
  direction      = "DEBIT"            ← 用户收款（钱进入用户）
  amount         = 30000
  entry_type     = "checkin"
  description    = "每日签到获得"

-- 复式记账守恒（全局永远成立）：
-- 全部账户的 SUM(DEBIT) - SUM(CREDIT) = 0
-- 系统: 0 DEBIT - 30000 CREDIT = -30000（发行方余额为负）
-- 用户A: 30000 DEBIT - 0 CREDIT = +30000（用户持有）
-- 全局: -30000 + 30000 = 0 ✓
```

再举一个投喂的例子——用户 A 投喂 5 小鱼干给文章，作者 B 获得 80%（4 小鱼干），系统回收 20%（1 小鱼干）：

```
-- 用户 A 支出 5 小鱼干（钱离开 A）
LedgerEntry 3: account=A, direction=CREDIT, amount=50000, type="feed_consume"
-- 作者 B 获得 4 小鱼干（钱进入 B）
LedgerEntry 4: account=B, direction=DEBIT, amount=40000, type="feed_income"
-- 系统回收 1 小鱼干（钱进入系统）
LedgerEntry 5: account=SYSTEM, direction=DEBIT, amount=10000, type="feed_income"

-- 验证：A支出 = 系统收 + B得
-- A: 50000 CREDIT 流出 = 支出 5
-- B: 40000 DEBIT 流入 = 收入 4
-- 系统: 10000 DEBIT 流入 = 回收 1
-- 全局: (-50000) + 40000 + 10000 = 0 ✓
```

### 2.5 余额计算

```sql
-- 实时余额查询（单个账户）
SELECT
    COALESCE(
        SUM(CASE WHEN direction = 'DEBIT' THEN amount ELSE -amount END),
        0
    ) AS balance
FROM ledger_entries
WHERE account_id = $1
  AND currency = 'DRIED_FISH';

-- 日终快照（可选，用于加速排行榜等高频查询）
-- 可以创建一个物化视图，每分钟或每小时刷新一次
CREATE MATERIALIZED VIEW account_balances AS
SELECT
    a.id AS account_id,
    a.user_id,
    a.currency,
    COALESCE(
        SUM(CASE WHEN le.direction = 'DEBIT' THEN le.amount ELSE -le.amount END),
        0
    ) AS balance
FROM accounts a
LEFT JOIN ledger_entries le ON le.account_id = a.id
GROUP BY a.id, a.user_id, a.currency;

-- 刷新物化视图（定时任务）
REFRESH MATERIALIZED VIEW CONCURRENTLY account_balances;
```

> 在用户规模 < 10 万时，实时聚合完全够用。`idx_ledger_account_id` 索引让单账户的 SUM 查询走 index-only scan。

## 3. 核心业务逻辑

### 3.1 转账（最关键的操作）

```python
# services/transfer_service.py — 实际流程（9 步）

class TransferService:
    def __init__(self, db: AsyncSession):
        self.db = db

    async def transfer(
        self,
        from_user_id: str,
        to_user_id: str,
        amount: Decimal,             # 外部单位，如 Decimal("5.0")
        currency: str,
        entry_type: str,
        description: str | None,
        metadata: dict | None,
        idempotency_key: str,
        api_key: str,                # 原始 API Key（来自 Authorization 头）
    ) -> TransferResponse:
        """
        原子转账。九步走：
        1. 检查幂等 → 2. 验证 API Key → 3. 转换金额
        → 4. 锁账户 → 5. 检余额 → 6. 写流水
        → 7. 算余额 → 8. 存幂等 → 9. 返回
        """

        # ── Step 1: 幂等检查 ──
        cached = await self._check_idempotency(
            idempotency_key, from_user_id, to_user_id,
            amount, currency, entry_type
        )
        if cached is not None:
            return cached  # 返回之前的响应

        # ── Step 2: 验证 API Key（在锁之前！）──
        key_hash = hash_api_key(api_key)
        from_account = await self._get_account_by_key_hash(key_hash)
        if from_account is None:
            raise ApiKeyInvalidError()
        if from_account.user_id != from_user_id:
            raise ApiKeyMismatchError(
                key_belongs_to=from_account.user_id,
                claimed_from_user=from_user_id,
            )

        # ── Step 3: 金额转换 ──
        internal_amount = to_internal(amount)  # 5.0 → 50000

        # ── Step 4: 锁账户（SELECT ... FOR UPDATE）──
        from_acc = await self._lock_account(from_account.id)
        to_acc = await self._get_or_create_account_for_transfer(
            to_user_id, currency
        )
        to_acc_locked = await self._lock_account(to_acc.id)

        # ── Step 5: 余额检查（跳过系统账户）──
        if not from_acc.is_system:
            balance_internal, _ = await self._compute_balance(from_acc.id)
            if balance_internal < internal_amount:
                raise InsufficientBalanceError(
                    user_id=from_user_id,
                    required=internal_amount,
                    available=balance_internal,
                )

        # ── Step 6: 写流水（CREDIT = 钱离开, DEBIT = 钱进入）──
        tx_id = uuid4()
        now = datetime.utcnow()

        credit_entry = LedgerEntry(
            transaction_id=tx_id,
            account_id=from_acc.id,
            direction="CREDIT",       # 发送方：钱离开
            amount=internal_amount,
            currency=currency,
            entry_type=entry_type,
            description=description,
            metadata_=metadata or {},
            created_at=now,
        )
        debit_entry = LedgerEntry(
            transaction_id=tx_id,
            account_id=to_acc_locked.id,
            direction="DEBIT",        # 接收方：钱进入
            amount=internal_amount,
            currency=currency,
            entry_type=entry_type,
            description=description,
            metadata_=metadata or {},
            created_at=now,
        )
        self.db.add_all([credit_entry, debit_entry])

        # ── Step 7: 计算转账后余额 ──
        from_balance_internal, _ = await self._compute_balance(from_acc.id)
        to_balance_internal, _ = await self._compute_balance(to_acc_locked.id)

        # ── Step 8: 保存幂等键 → 响应映射 ──
        response = TransferResponse(
            transaction_id=tx_id,
            from_user_id=from_user_id,
            to_user_id=to_user_id,
            amount=amount,
            currency=currency,
            entry_type=entry_type,
            from_balance_after=to_external(from_balance_internal),
            to_balance_after=to_external(to_balance_internal),
            created_at=now,
        )
        await self._save_idempotency(idempotency_key, tx_id, response)

        # ── Step 9: 返回（commit 由 FastAPI 依赖 get_db 处理）──
        return response
```

### 3.2 余额查询

```python
# services/account_service.py

class AccountService:
    async def get_balance(self, user_id: str, currency: str = "DRIED_FISH") -> Decimal:
        """实时聚合余额。对单个账户的 SUM 查询，毫秒级完成。"""
        result = await self.db.execute(
            select(
                func.coalesce(
                    func.sum(
                        func.case(
                            (LedgerEntry.direction == 'DEBIT', LedgerEntry.amount),
                            else_=-LedgerEntry.amount,
                        )
                    ),
                    0,
                )
            )
            .join(Account, LedgerEntry.account_id == Account.id)
            .where(Account.user_id == user_id)
            .where(LedgerEntry.currency == currency)
        )
        raw_balance = result.scalar_one()
        return to_external(raw_balance)  # 50000 → Decimal("5.0")

    async def get_balance_batch(
        self, user_ids: list[str], currency: str = "DRIED_FISH"
    ) -> dict[str, Decimal]:
        """批量查询余额。一次 SQL 返回所有结果。"""
        result = await self.db.execute(
            select(
                Account.user_id,
                func.coalesce(
                    func.sum(
                        func.case(
                            (LedgerEntry.direction == 'DEBIT', LedgerEntry.amount),
                            else_=-LedgerEntry.amount,
                        )
                    ),
                    0,
                ).label('balance')
            )
            .join(LedgerEntry, LedgerEntry.account_id == Account.id, isouter=True)
            .where(Account.user_id.in_(user_ids))
            .where(LedgerEntry.currency == currency)
            .group_by(Account.user_id)
        )
        rows = {row.user_id: to_external(row.balance) for row in result}
        # 填充未找到的 user_id 为 0
        return {uid: rows.get(uid, Decimal("0")) for uid in user_ids}
```

### 3.3 精度转换

```python
# core/currency.py

# 1 小鱼干 = 10000 内部单位，精度到 0.0001
INTERNAL_SCALE = 10000

def to_internal(external: Decimal | float | int) -> int:
    """外部单位 → 内部整型，四舍五入"""
    return round(Decimal(str(external)) * INTERNAL_SCALE)

def to_external(internal: int) -> Decimal:
    """内部整型 → 外部 Decimal"""
    return Decimal(internal) / INTERNAL_SCALE
```

> **为什么用 BIGINT 而不是 NUMERIC？** 对于虚拟货币，固定精度整型（1 小鱼干 = 10000）比数据库的 NUMERIC/DECIMAL 更快。SUM 聚合在 BIGINT 上是原生 CPU 指令，NUMERIC 要走软件实现。而且 1 小鱼干的精度（0.0001）对于任何虚拟货币场景都够用了。


博客系统的特征：
- `is_system = TRUE` — 转账时跳过余额检查（可无限透支），作为鱼干的"发行方"
- `user_id = 'raricy-blog-system'` — 特殊的 `user_id`，不与任何博客用户冲突
- 初始 API Key 写入 `.env` 文件，部署后应立即轮换

**博客系统账户的 API Key 存储在博客的 `.env` 中：**

```ini
# 博客 .env
ACCOUNT_SERVICE_URL=http://account-service:8000
ACCOUNT_SYSTEM_KEY=fish_sk_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

这意味着：博客系统和普通用户在账户服务眼中完全平等——都通过 `Authorization: Bearer <key>` 认证。唯一的区别是博客系统的账户有 `is_system = TRUE`，允许余额为负（透支发行鱼干）。
