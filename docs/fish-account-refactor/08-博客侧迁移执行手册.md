## 2. 配置 .env

在博客项目的 `.env` 文件中添加以下配置：

```ini
# 账户服务地址
ACCOUNT_SERVICE_URL=http://localhost:8000
# 系统账户的 API Key（seed.py 输出的）
ACCOUNT_SYSTEM_KEY=fish_sk_your_system_api_key_here
# 与账户服务 INTERNAL_TOKEN 一致的共享密钥
ACCOUNT_SERVICE_INTERNAL_TOKEN=your_shared_internal_token_here
# 请求超时（秒）
ACCOUNT_SERVICE_TIMEOUT=5
```

**生成 INTERNAL_TOKEN**（两端需一致）：
```bash
python -c "import secrets; print(secrets.token_urlsafe(32))"
```

## 7. 数据迁移（已完成部分）

> **2026-07-08 策略重构**：废弃 `--replay-transactions`（逐条回放历史 FishTransaction），改为 `--sync-balances` 全量余额迁移。详见 §7.5 设计说明。

### 7.1 全量余额迁移（`--sync-balances`）

```bash
python scripts/migrate_fish_data.py --sync-balances
```

对每个用户，将其当前的 `User.dried_fish` 一次性 `SYSTEM → user` 转账到远端账户。

- **金额**：`user.dried_fish`（已包含所有历史交易的净结果）
- **entry_type**：`migration_init`
- **幂等键**：`migrate-init-{user_id}`，重跑不会双倍转账
- **跳过条件**：`user.dried_fish <= 0.01` 直接跳过

> 重构前的旧版 `--sync-balances` 是计算"本地余额 - 可回放流水总额"的差额补齐（仅补初始余额），依赖 `--replay-transactions` 先跑。新版直接搬运全量余额，不再依赖流水回放。

### 7.2 修复差异（`--fix-discrepancies`）

```bash
python scripts/migrate_fish_data.py --fix-discrepancies
```

对每个用户对比本地 `dried_fish` 与远程余额，若 `|差异| > 0.01` 调用 transfer 补齐：

- `local > remote`：`SYSTEM → user` 增发
- `local < remote`：`user → SYSTEM` 回收
- entry_type = `migration_adjust`
- 幂等键 = `migrate-adjust-{user_id}-{±cents}`，相同 diff 重复执行不会双倍调整

**重要**：修复前必须先排查差异来源（对比 FishTransaction 和 ledger_entries），确认差异不是来自"真实但未同步"的双写失败。

### 7.3 当前差异（2026-07-04 实测）

2026-07-04 跑过 `--sync-balances`（旧版）后仍有 2 个用户余额漂移：

| 用户 | 本地 | 远程 | 差异 |
|------|------|------|------|
| 123123 | 20.40 | 17.60 | +2.80（本地多） |
| 234234 | 2.00 | 3.00 | −1.00（远程多） |

**差异原因**：原 Phase 1 best-effort 双写被吞的历史失败流水。例如：
- 投喂 → 本地 deduct 成功 → 远端 `feed_transfer` 失败 → 本地 -amount，远程 0（本地多）
- 签到 → 本地 add 成功 → 远端 transfer 失败 → 本地 +amount，远程 0（本地多）
- 反向也可能（罕见，比如本地事务被回滚但远端已成功）

已于 2026-07-04 用 `--fix-discrepancies` 修复这两个用户。

```bash
python scripts/migrate_fish_data.py --all                 # 开户 + 全量余额迁移
python scripts/migrate_fish_data.py --fix-discrepancies   # 单独修复差异（按需）
```

### 7.5 迁移策略设计说明（2026-07-08）

**重构前的问题**：

旧版 `--replay-transactions` 把 `fish_transactions` 表里**每条历史记录**都创建一笔远端 ledger 条目。结果是：
- 用户的 100+ 条历史签到都被压到迁移当天同一秒 → 远端 ledger 大量"伪造"的 `entry_type='checkin'` 条目，`created_at` 全部是迁移日
- `_sum_today_checkin` 等"按时间窗统计签到获得鱼干"的查询语义失真
- 迁移脚本运行时间长、节流开销大

**重构后的设计**：

签到记录（`daily_check_ins`）和鱼干账本（账户服务 ledger）是两个独立维度：

| 维度 | 责任 | 存储 |
|------|------|------|
| 用户哪天签到 / 抽到什么卡 | 博客业务 | `daily_checkins` 表 |
| 鱼干从哪来（签到/投喂/管理） | 鱼干系统 | 账户服务 ledger（按 entry_type 分类） |

迁移只做一件事：把每个用户当前的 `User.dried_fish` 全量转账到远端账户（一条 `migration_init` 流水）。`User.dried_fish` 已经是所有历史交易（签到、admin_grant、feed、purchase 等）的净结果，全量迁移等于"完整还原历史账目"。

之后：
- 每次签到 → 写 `daily_checkins`（博客业务）+ 一条 `entry_type='checkin'` 的 ledger，`created_at` 是真实业务时间
- 每次投喂 → 走账户服务 `feed_transfer`，自动产生 `feed_consume` + `feed_income` 两条 ledger
- 每次管理发放 → 一条 `entry_type='admin_grant'` 的 ledger

`fish_transactions` 表保留，作为本地审计/远端故障 fallback（`get_today_checkin_fish` 等函数仍在用），但不再是 source of truth。Phase 3 清理时再决定是否物理删除。

## 10. 错误注入测试

> **本节反映 Phase 1.5 之后的统一行为**：所有写路径（投喂/签到/CLI grant/CLI deduct）均 fail-closed。远端不可用时，本地零变更，用户收到明确错误。

**测试 A：远端挂掉时所有写路径应失败（fail-closed 验证）**

1. **停掉账户服务**：`pkill -f uvicorn`
2. **投喂 API** → 应返回 **HTTP 503** "鱼干服务暂不可用，请稍后再试"
3. **签到 API** (claim-fortune) → 应返回 **HTTP 503**
4. **CLI**：`flask fish grant <user> 5` → 应**失败退出（exit 2）**，本地余额不变
5. **CLI**：`flask fish deduct <user> 5` → 应**失败退出（exit 2）**，本地余额不变
6. 检查数据库：所有 `User.dried_fish`、`Blog.fish_count`、`User.total_fortune`、`FishTransaction` 均**不应有任何变更**（rollback 生效）
7. 检查博客日志：`账户服务.*同步失败，本地事务已回滚`

**测试 B：远端恢复后功能正常**

1. **重启账户服务**：`cd account-service && uvicorn app.main:app --port 8000 &`
2. 投喂 → 恢复 200
3. 签到 → 恢复 200
4. CLI grant/deduct → 恢复成功

## 13. 故障排查

| 现象 | 可能原因 | 解决方法 |
|------|----------|----------|
| `AccountClientError: 账户服务不可达` | 账户服务未启动 | `cd account-service && uvicorn app.main:app --port 8000` |
| `Internal token invalid` | INTERNAL_TOKEN 不匹配 | 检查两边的 .env，确保一致 |
| `API key not found` | 系统 API Key 错误 | 重新 seed 并更新 ACCOUNT_SYSTEM_KEY |
| 注册后 `fish_api_key_encrypted` 为空 | 账户服务创建失败 | 检查日志；手动运行 `migrate_fish_data.py --create-accounts` |
| 对账有差异 | 双写失败或迁移未完成 | 先确认 `--all` 已跑完，再检查日志；持续差异跑 `--fix-discrepancies` |
| 投喂超时 | 网络延迟或服务过载 | 调整 `ACCOUNT_SERVICE_TIMEOUT`；**投喂已 fail-closed**：超时会让本次投喂直接失败（503），本地零变更 |
