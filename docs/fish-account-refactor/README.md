## 术语表

| 术语 | 说明 |
|------|------|
| **小鱼干** | raricy.com 的虚拟货币，通过签到、被投喂等途径获得。当前 1 小鱼干 = 10000 内部存储单位 |
| **FISH_UNIT** | 当前代码中的精度常量（= 10000）。新账户服务中将隐藏在服务端，调用方不需要感知 |
| **复式记账 (Double-Entry)** | 每笔交易的金额同时记录借方 (DEBIT) 和贷方 (CREDIT)，余额从流水聚合计算而非直接存储 |
| **幂等 (Idempotency)** | 同一个请求无论执行多少次，结果相同。通过 `X-Idempotency-Key` 实现，防止网络重试导致重复扣款 |
| **服务间鉴权** | 双层认证：`X-Internal-Token` 共享密钥（所有端点）+ API Key（转账端点）。账户服务不直接暴露给公网 |
| **API Key** | 每个账户（包括博客系统）有独立的 `fish_sk_` 前缀 API Key。转账时提供 `from_user` 的 Key 作为 Bearer token。没有特权通道 |

## 核心设计原则

1. **余额是派生的，不是存储的** — `balance = SUM(ledger_entries)`，不可篡改
2. **每笔交易可审计** — 任意时间点都能回溯到当时的余额
3. **博客不知道小鱼干的内部实现** — 博客只调 API，不关心精度单位、存储方式
4. **为扩展而生** — `currency` 字段预留多币种；`LedgerEntry` 的 `metadata` JSON 字段可承载各种业务上下文
5. **服务间双向鉴权** — 所有请求需 `X-Internal-Token` 共享密钥（验证调用方身份）；资金操作额外需 API Key（验证用户身份）。纵深防御，各司其职
