## 权限对齐说明（已按 Flask 逐条核定）
- `/blog/upload`、`/blog/[id]/edit`：`@authenticated_required`（core+）→ 未登录/非核心 403（实测 403）。
- `/fish`、`/fish/transactions`：`@login_required`（任意登录用户）→ 未登录 307 跳登录（实测 307）。

- 博客编辑：Flask 为**仅作者本人**（admin 也不可），已忠实复刻。

## 🔴 上线前仍需满足的运行时前置条件（部署项，非代码缺口）

> **先跑自检**：`cd web-next && npm run diagnose -- --url https://你的域名`
> 它会把下列 1/2/8/9 及权限、路径问题一次性指出来，不用逐条人肉核对。

1. **时间戳格式**：真实库需先跑 `scripts/normalize-datetimes.mjs` 转 **INTEGER（Unix 毫秒）**。
   · 不转 → Prisma 在驱动层解析 DATETIME 即抛 `Conversion failed`（P2023）→ **登录必 500**。
   · 转成 TEXT ISO（旧版脚本的产物）→ 能读，但**日期比较会按 SQLite 类型序而非数值**
     （gte 恒真 / lt 恒假）→ 发文日限额把历史文章全算成「今天」，历史发文 ≥20 篇的用户
     永久发不了文。详见 [03 §3](03-数据库映射与陷阱.md)。`npm run diagnose` 能分辨这三种状态。
   **⚠️ 规整不可逆，且规整后 Flask 读不了该库** —— 若新旧站共用同一个库，先读
   [03 §3「必须知道的取舍」](03-数据库映射与陷阱.md)，别直接就地规整。见下方「切换路线」。
2. **经济系统生产密钥**：`.env.production` 必须填入 Flask 生产环境实际加密
   `fish_api_key_encrypted` 用的密钥（本仓库 `.env` 的 SECRET_KEY 解不开真实数据）。
   否则 465 个用户鱼干账户解不开、经济系统 fail-closed(503)。**这是 Flask 同样具备的依赖。**
3. **账户微服务可达且已联调**：TS 客户端 + Fernet 兼容已验，但尚未对运行中的真实
   account-service 端到端联调（本环境无法起该服务）。切换前需真连跑通投喂/发鱼一次。
4. **data/ 目录随部署提供**：头像（data/avatars）、图片（data/images）由 Next 原生读取，
   通过 `.env` 的 `AVATARS_DIR` / `IMAGE_UPLOAD_FOLDER` 指向。部署时挂载真实数据卷。
5. **会话不互通**：切换后用户需重新登录一次（密码不用改，session_version 从 0 起）。
6. **内存限频**：单进程有效；多实例部署需换 Redis。
7. **SQLite 并发写**：高并发写是 SQLite 固有短板，长期建议迁 Postgres（[03 §7](03-数据库映射与陷阱.md)）。
8. **反向代理必须透传原始 Host / 协议**（实测踩坑）：nginx 需配
   `proxy_set_header Host $host` / `X-Forwarded-Host $host` / `X-Forwarded-Proto $scheme`，
   否则：① 正常写请求被 CSRF 中间件误判为跨源 → 403；② 会话 cookie 的 Secure 判定失准。
   反代配置不确定时，直接在 `.env.production` 设 `ALLOWED_ORIGINS="你的域名"` 兜底。
   见 [02「部署在 nginx 反向代理后面」](02-快速开始.md)。
9. **TLS / 会话 cookie**（实测踩坑）：站点走纯 HTTP 时，带 `Secure` 的 cookie 会被浏览器丢弃
   →「登录返回 200 成功、但刷新仍未登录，且无任何报错」。已改为按 `X-Forwarded-Proto` 自动判定，
   nginx 透传了协议即自动正确；否则显式设 `COOKIE_SECURE="false"`。
   **⚠️ HTTP 下会话 cookie 明文传输、可被链路窃取冒用 —— 生产站点应上 TLS**，
   `COOKIE_SECURE=false` 仅作过渡期的显式选择。（原 Flask 未设 `SESSION_COOKIE_SECURE`，故老站 HTTP 下可用。）

## 切换路线：先选一条（时间戳规整是一扇单向门）

**关键约束**：同一个库**不能**同时让 SQLAlchemy 和 Prisma 满意。规整成 INTEGER 毫秒后 Flask 就读不了。
因此「新旧站并行灰度」**不是免费的**，必须先选路线：

| 路线 | 做法 | 能否并行灰度 / 回滚 | 适用 |
|---|---|---|---|
| **A. 硬切换** | 停 Flask → 备份 → `--in-place` 规整线上库 → 起 Next | ❌ 不能并行；回滚需还原备份 | 接受停机窗口、一次到位 |
| **B. 兼容层灰度**（推荐） | 先给 Flask 上 [03 附录 A](03-数据库映射与陷阱.md) 的 INTEGER 毫秒兼容 `TypeDecorator` → 规整线上库 → 两边读同一个库 | ✅ 可并行、可秒切回 Flask | 想稳妥灰度、留退路 |
| **C. 副本验证** | `--source/--dest` 复制一份规整后的库给 Next 用，原库不动 | ⚠️ 数据从此分叉，仅供验证 | 功能验收，**别放真实用户** |

> 路线 C 期间，Next 上产生的注册/发帖**不会回到老站**，正式切换时会丢——测试期请勿对外宣传。

## 🔧 切换前必须执行的数据补偿

### 「已签到但未翻牌」的记录（fortune_value IS NULL）

Flask 的签到是**两步**（check_in 建记录 → claim_fortune 翻牌发鱼），Next 合并成了一步，
**没有单独的 claim 入口**。因此库里存量的「已签到未翻牌」记录，切换后这些用户
永远翻不了那张牌（doCheckin 会被唯一约束挡住），当天的鱼干与 total_fortune 永久丢失。

真实库里有 **7 条**（最近一条 2026-07-15）。切换前跑：

```bash
cd web-next
node scripts/compensate-unclaimed-fortunes.mjs --db <目标库>            # 预演，先看一遍
node scripts/compensate-unclaimed-fortunes.mjs --db <目标库> --apply    # 执行
```

补偿口径逐条对齐 Flask `claim_fortune`：从该记录**自己的** `fortune_pool` 里取一张
（池是签到当时就定好的）→ 原子置 fortune_value → `total_fortune += value` →
发等额鱼干 + 写 `type='checkin'` 流水。默认 dry-run，必须显式 `--apply`；
每步都是原子条件写（`WHERE fortune_value IS NULL`），**可重复执行不会二次发鱼**。

> ⚠️ 该脚本只写本地库。**必须在切换前（远端账户服务尚未接管时）执行**，
> 否则补发的鱼干需要额外与账户微服务对账。

## 删除 Flask 的操作顺序（建议）
1. 满足上方全部前置条件（先跑 `npm run diagnose` 自检）。
   **装依赖一律用 `npm ci`**（严格按 lockfile），不要用 `npm install next@latest` /
   `npm update` —— 实际发生过跑成 Next 16 导致启动即崩溃（本项目对着 Next 15 写）。
2. **选定切换路线**（A/B/C）。若要并行灰度，必须走 B —— 先上 03 附录 A 的兼容层。
3. **跑数据补偿**：`node scripts/compensate-unclaimed-fortunes.mjs --db <线上库> --apply`
   （见上方「切换前必须执行的数据补偿」）。
4. 部署 web-next 到生产；冒烟 + 验证所有模块（尤其经济系统真连账户服务）。
5. 切流量到 Next；观察一段时间（走 B 的话此时可随时切回 Flask）。
6. 确认无回退需求后删除 Flask。**清单见 [07 §6](07-切换操作手册.md)** —— 那里逐项
   写明了「删什么」和「千万别跟着删什么」。

   > ⚠️ 这里原先写的是「删除 `app/`、`run.py`、`account-service/` 的 Flask 侧代码」。
   > 那句话有歧义且危险：**`account-service/` 是 Next 仍在使用的 FastAPI 微服务**
   > （checkin / feed / fish 三条写路径都调它），删掉它全站鱼干立刻 fail-closed 503。
   > 它从来就不是 Flask 的一部分，不在删除之列。
