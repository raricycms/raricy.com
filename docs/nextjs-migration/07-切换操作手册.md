# 07 · 切换操作手册

把 raricy.com 从 Flask 切到 Next.js 的**实际执行步骤**。前面 6 篇讲「为什么这么设计」，这篇只讲「照着敲什么」。

**执行者**：有服务器 root / nginx / systemd 权限的人。
**预计耗时**：路线 A 约 30–50 分钟（含停机 10–20 分钟）。

> **本手册已在真实数据副本上完整彩排过**（465 用户 / 6183 文章 / 63209 评论的
> 生产库副本），下面每条命令都是实跑验证的，不是照着代码推想出来的。彩排结论见 §5。

---

## 0. 先决条件（不满足就别开始）

| # | 条件 | 怎么确认 |
|---|------|----------|
| 1 | 服务器有 Node ≥ 20 | `node -v` |
| 2 | 用 `npm ci` 装依赖，**不是** `npm install` | 见下方「⚠️ Next 版本」 |
| 3 | 账户微服务（account-service）已部署且能通 | `curl $ACCOUNT_SERVICE_URL/health` |
| 4 | **拿到 Flask 生产的 `SECRET_KEY`**（且 `FISH_ENCRYPTION_KEY` 留空） | 见 §1 —— 这条最容易漏，漏了 465 个用户的鱼干全废 |
| 5 | 数据库已备份，且备份**验证过能打开** | `sqlite3 backup.db "select count(*) from users"` |
| 6 | 选好路线（A 硬切 / B 灰度） | 见 §2 |

### ⚠️ Next 版本必须锁 15.5.20

`npm install` 会把 next 升到 16.x，而 16 与本项目不兼容，症状是启动即崩：

```
TypeError: Cannot read properties of undefined (reading 'map')
    at ignore-listed frames
```

**永远用 `npm ci`** —— 它严格按 package-lock.json 装。已经装错了就：

```bash
cd web-next
rm -rf node_modules
npm ci                  # ← 别加 --omit=dev，见下
npx next --version      # 必须输出 15.5.20
```

诊断脚本会替你查这一条：`npm run diagnose`。

> **别用 `npm ci --omit=dev`。** 运维要用的 `npm run diagnose`、`npm run cli`
> （升降权限、发/扣鱼干）都靠 `tsx` 跑，而它在 devDependencies 里。省掉 dev 依赖
> 会让这些命令直接找不到 —— 而它们恰恰是出事时你最需要的。

## 1. 生产 FISH 加密密钥（最容易漏的一步）

**背景**：每个用户在账户微服务里有一个 API Key，用 Fernet 加密存在
`users.fish_api_key_encrypted`。Flask 的密钥**只**从 `SECRET_KEY` 派生
（`app/clients/account_client.py:93`）：

```
Fernet(base64.urlsafe_b64encode(sha256(SECRET_KEY)))
```

**这意味着**：`SECRET_KEY` 一变，全库已加密的 API Key 全部解不开 —— 465 个用户
的鱼干功能集体失效，且**不可逆**（密文还在，但没有密钥）。

所以：

- 生产的 `SECRET_KEY` 必须与**当前 Flask 生产环境正在用的那个**完全一致。
  它在服务器的 `.env` 里，**不在本仓库**（仓库里的是开发用的，解不开任何生产密文）。
- 切换时把它原样搬进 `web-next/.env`，别重新生成。
- 顺手确认它没被提交进 git：`git log -p --all -S "$SECRET_KEY" | head` 应无输出。

**验证密钥对不对**（切换前就能验，别等上线被用户投诉）：

```bash
cd web-next
DATABASE_URL="file:/path/to/prod-normalized.db" SECRET_KEY="<你要用的那个>" npm run diagnose
```

`diagnose` 的第 4 节会拿库里 5 条真实密文试解，直接告诉你钥匙对不对：

```
4. 小鱼干密钥（切换前必查，错了不可逆）
  ✓ FISH_ENCRYPTION_KEY 留空（回退 SECRET_KEY，与 Flask 一致）
  ✓ SECRET_KEY 正确：抽查 5/5 条存量密文全部解开
```

钥匙不对会是这样，并直接点出原因：

```
  ✗ SECRET_KEY 不对：抽查 5 条，只解开 0 条（Invalid Token: HMAC）
     → SECRET_KEY 必须与当前 Flask 生产环境用的完全一致 —— 它在服务器的 .env 里，不在仓库
```

**这是整个切换里唯一不可逆的一步，也是唯一能提前机械验证的一步。** 别跳过。

### ⚠️ `FISH_ENCRYPTION_KEY` 必须留空

Next 侧的派生源是 `FISH_ENCRYPTION_KEY || SECRET_KEY`，而 **Flask 没有前者** ——
库里所有存量密文都是 `SECRET_KEY` 加密的。

一旦在生产 `.env` 里给 `FISH_ENCRYPTION_KEY` 填了值，Next 就改用它派生密钥，
存量密文全部解不开（实测报 `Invalid Token: HMAC`），全站鱼干功能集体失效。

**留空即可** —— 回退 SECRET_KEY，与 Flask 一致。只有全新部署、库里没有任何
存量密文时，才谈得上设独立密钥。

## 2. 选路线

### 路线 A · 硬切换（推荐）

```
停 Flask → 备份 → 规整时间戳 → 补偿 → 起 Next → 切流量
```

- **优点**：一次到位，不用维护双写，没有「两套栈同时写库」的一致性风险。
- **缺点**：停机 10–20 分钟；回滚 = 恢复备份（期间的新数据会丢）。
- **适合**：站点流量不大、能接受一个凌晨的停机窗口。**本项目建议走这条。**

### 路线 B · 灰度（可回滚）

```
停 Flask → 备份 → 规整 → 给 Flask 装兼容层 → 两套栈共读一个库 → 逐步切流量
```

- 关键在 `app/extensions/ms_datetime.py`：规整之后 Flask 就读不了自己的库了
  （`fromisoformat: argument must be str`），装上这层它才能继续跑。
  开启方式：Flask 侧设 `DB_DATETIME_MS=true`（默认关闭）。
- **优点**：出事能把 nginx 切回 Flask，不丢数据。
- **缺点**：一段时间内两套栈同时写同一个 SQLite —— SQLite 的写锁是库级的，
  并发写会互相 `database is locked`。**只在你确实需要回滚能力时用。**

## 3. 执行步骤（路线 A）

### 3.1 停 Flask

```bash
sudo systemctl stop raricy        # 你的 Flask service 名
sudo systemctl status raricy      # 确认真停了
```

> 必须先停。库还在被写的时候做规整，会丢掉规整开始之后写入的那些行。

### 3.2–3.4 一条命令搞定（推荐）

下面 3.2 备份 / 3.3 规整 / 3.4 补偿这三步**有严格的先后依赖，且每步都得验完才能走
下一步**。凌晨的维护窗口里靠人记住这些顺序不现实，所以串成了一条命令：

```bash
cd web-next

# 预演：只读源库，产出规整后的新库，补偿只打印不写
SECRET_KEY="<生产的>" npm run prepare:cutover -- \
  --source /path/to/instance/database/db.db \
  --dest   /path/to/prod-normalized.db

# 看完上面逐条列出的补偿变更，确认无误后执行
SECRET_KEY="<生产的>" npm run prepare:cutover -- \
  --source /path/to/instance/database/db.db \
  --dest   /path/to/prod-normalized.db --apply
```

它会依次做完并逐项验证：

| 步骤 | 验什么 |
|------|--------|
| 0 | 源库是否还有活跃 WAL（Flask 没停就规整会**静默丢数据**） |
| 1 | `.backup` 取备份，并验证备份能打开、行数与源库一致 |
| 2 | 规整时间戳，验证行数无损失 |
| 3 | **33 个时间列逐条比对墙上时间，一处漂移都不许有** |
| 4 | 补偿未翻牌的签到（默认只预演，逐条打印变更） |
| 5 | 跑 diagnose，含 **SECRET_KEY 能否解开存量密文**（§1 那条不可逆的坑） |
| ✓ | 比对源库 SHA-256，证明全程没碰过它 |

任何一项没过就 `exit 1`，不会让你稀里糊涂往下走。

> **目标库已存在时它会拒绝执行**，除非你加 `--overwrite-dest`。因为第 2 步会把目标库
> 推平重建 —— 切换**之后**网站就跑在那个库上了，那时手滑重跑一次，上线后新增的
> 一切都会消失，而源库里没有它们。

下面 3.2–3.4 是这条命令内部做的事，想手动来、或想搞清楚它在干嘛时看。

### 3.2 备份（做完立刻验证）

```bash
D=~/backup/$(date +%Y%m%d-%H%M%S)
mkdir -p $D
sqlite3 /path/to/instance/database/db.db ".backup $D/db.db"

# 验证备份可用 —— 别跳过这步，坏备份等于没备份
sqlite3 $D/db.db "select count(*) from users; select count(*) from blogs;"
```

> 用 `.backup` 而不是 `cp`：`cp` 在有 WAL 时会拷到不一致的快照。

### 3.3 规整时间戳

**这是整个迁移的核心一刀。** Prisma 往 SQLite 写 DateTime 用 INTEGER（Unix 毫秒），
SQLAlchemy 写的是 `2026-07-16 10:00:00.123456` 文本。同一列两种格式，Prisma 读到
文本就直接 500：

```
Inconsistent column data: Conversion failed: input contains invalid characters
code: 'P2023'
```

脚本把全库时间戳统一成 INTEGER 毫秒：

```bash
cd web-next
node scripts/normalize-datetimes.mjs \
  --source /path/to/instance/database/db.db \
  --dest   /path/to/prod-normalized.db
```

**它不改源库**，只读源、写目标 —— 源库始终是你的兜底。

时区语义：转换**不做时区平移**，墙上时间原样保留。本库的既有约定是
「naive UTC+8 墙上时间」（Flask 用 `datetime.now()` 写服务器本地时间，服务器
TZ=UTC+8）。这个结论是从真实数据反推证实的：`daily_checkins` 里显式按 UTC+8
算出来的 `checkin_date` 与 `date(created_at)` **2170/2170 全等**；若服务器是 UTC，
UTC 16:00–23:59 那 548 条签到必然跨日不等。详见 [03-数据库映射与陷阱](03-数据库映射与陷阱.md)。

规整完验一下：

```bash
sqlite3 /path/to/prod-normalized.db \
  "select distinct typeof(created_at) from users;"   # 应只输出 integer
```

### 3.4 补偿未翻牌的签到

历史上有 7 条签到记录 `fortune_value` 为 NULL（用户签了但没翻牌），新系统
不认这种半成品状态。补偿脚本**默认 dry-run**，先看一遍再执行：

```bash
# 预演 —— 逐条打印将要发生的变更，不写库
node scripts/compensate-unclaimed-fortunes.mjs --db /path/to/prod-normalized.db

# 确认无误后执行
node scripts/compensate-unclaimed-fortunes.mjs --db /path/to/prod-normalized.db --apply
```

脚本是幂等的（`WHERE fortune_value IS NULL`），重复跑不会重复发鱼干。

### 3.5 配 .env

```bash
cd web-next
cp .env.example .env      # 里面每个变量都有注释，照着填
```

关键的几个：

| 变量 | 值 | 填错会怎样 |
|------|-----|-----------|
| `DATABASE_URL` | `file:/path/to/prod-normalized.db` | 起不来 |
| `SECRET_KEY` | **与 Flask 生产完全一致**（见 §1） | 465 个用户鱼干全废 |
| `FISH_ENCRYPTION_KEY` | **留空**（见 §1） | 同上 |
| `ACCOUNT_SERVICE_URL` | 账户服务地址 | 见下 |
| `ACCOUNT_SERVICE_INTERNAL_TOKEN` | 与账户服务的 `INTERNAL_TOKEN` 一致 | **生产环境所有鱼干写路径直接 503**（fail-closed，故意的） |
| `ACCOUNT_SYSTEM_KEY` | 账户服务 `seed.py` 输出的系统 Key | 同上 |
| `ALLOWED_ORIGINS` | `https://raricy.com`（逗号分隔多个） | 见下方 CSRF |
| `COOKIE_SECURE` | `true`（走 HTTPS 时） | 见下方 cookie |
| `NODE_ENV` | `production` | fail-closed 不生效 |

**关于 `ACCOUNT_SERVICE_INTERNAL_TOKEN`**：不配的话，生产环境下投喂/签到/注册/
CLI 发鱼干会**直接报 503 而不是静默跳过远端**。这是设计如此（CLAUDE.md Phase 1.5
fail-closed）—— 宁可功能不可用，也不能出现「本地扣了鱼干但远端没记账」的账目撕裂。

### 3.6 装依赖 + 构建 + 自检

```bash
npm ci                    # 不是 npm install（见 §0）
npm run build
npm run diagnose          # 版本 / .env / 库格式 全套自检
```

`diagnose` 会查：Next 大版本、必需环境变量、数据库文件、时间戳类型是否已规整、
**小鱼干密钥能否解开存量密文**（§1 那条不可逆的坑），以及（可选）对线上地址做一次
活体登录检查。**它报红就别往下走。**

账户服务的连通性它**不查**，单独验：

```bash
curl -s $ACCOUNT_SERVICE_URL/health
# 端到端对账（需一个独立空库，别指真实库）
npm run verify:account -- --db /tmp/verify.db
```

### 3.7 起服务

先前台试一把，确认能起来：

```bash
npm run start
curl -I http://127.0.0.1:3000/
```

然后装成 systemd 服务 —— **别就这么前台挂着**：终端一关站就没了，服务器重启也不会
自己回来，而且没人会立刻发现。

```bash
sudo cp docs/nextjs-migration/raricy-next.service.example /etc/systemd/system/raricy-next.service
sudo vim /etc/systemd/system/raricy-next.service    # 改 User / WorkingDirectory / ExecStart 的路径
sudo systemctl daemon-reload
sudo systemctl enable --now raricy-next             # enable = 开机自启
systemctl status raricy-next
journalctl -u raricy-next -f                        # 看日志
```

[raricy-next.service.example](raricy-next.service.example) 里写了两条容易踩的：

- **别配 `EnvironmentFile=.env`** —— `next start` 自己会读同目录的 `.env`
  （实测：shell 环境清空到只剩 `PATH`，站点照常起来）。再挂一个 `EnvironmentFile`
  就是第二个真相源，而 systemd 解析 env 文件的规则与 dotenv 不一致（引号、`#` 的
  处理都不同）—— 值里有特殊字符时两边会读出不同的东西，而 `SECRET_KEY` 恰恰
  最可能有特殊字符。
- **`NODE_ENV` 不用设** —— `next start` 本身就是 production（实测：未配账户服务时
  注册返回 503，正是 production 下 fail-closed 生效的证据）。

### 3.8 切 nginx

**最省事、最不容易出错的做法：别重写现有配置，只把 `proxy_pass` 指向 3000,其它
一个字别动。** 你现有的配置既然在跑 Flask 的图床，那 `client_max_body_size` 之类
本来就都配好了。

要新写的话，用 [nginx.conf.example](nginx.conf.example) —— 那份是完整的，且本地起
真 nginx 逐条验过（2MB 上传穿透、12MB 被拦、外站 Origin 403、同源放行、头像回源、
静态资源长缓存）。下面只讲最容易漏、漏了必出事的几条：

```nginx
client_max_body_size 12m;          # ← 少了它：图床只能传 1MB 以下的图

location / {
    proxy_pass http://127.0.0.1:3000;
    proxy_set_header Host              $http_host;   # ← 用 $http_host，别用 $host
    proxy_set_header X-Forwarded-Host  $http_host;   # ← 少了它：CSRF 403
    proxy_set_header X-Forwarded-Proto $scheme;      # ← 少了它：登录后没登录态
    proxy_set_header X-Real-IP         $remote_addr;
}
```

- **`client_max_body_size`**：nginx 默认只给 **1MB**，而图床单文件上限是 10MB。
  不配的话超过 1MB 的图会被 nginx 直接回 **413** —— 请求**根本到不了 Next**，
  应用日志里干干净净，排查时会一头雾水。（本地实测：注释掉这行，2MB 上传立刻 413。）
- **`$http_host` 而不是 `$host`**：`$host` 不含端口，而中间件比对的是含端口的 host。
  走标准 443 时两者都没端口、碰巧不出事；但站点若在非标准端口上，`$host` 会让全站
  POST 403。（本地实测：`$host` + 8443 → 403，换 `$http_host` → 通过。）

为什么这两个头是致命的：

- **`X-Forwarded-Host`**：中间件拿请求的 `Origin` 跟允许列表比对。套了 nginx 之后
  `Host` 是 `127.0.0.1:3000`，而浏览器发来的 `Origin` 是 `https://raricy.com`，
  对不上 → 所有 POST 被判成 CSRF → **403**。
- **`X-Forwarded-Proto`**：Next 靠它判断要不要给 cookie 加 `Secure`。判断错了，
  cookie 会被浏览器丢掉，表现为「登录接口返回成功，但刷新还是未登录」——
  这个症状极具迷惑性，因为登录本身是成功的。

```bash
sudo nginx -t && sudo systemctl reload nginx
```

### 3.9 冒烟测试（按顺序，全过才算切成功）

| # | 动作 | 期望 |
|---|------|------|
| 1 | 打开首页 | 导航页正常渲染（**首页本就不列文章**，别在这找文章） |
| 2 | 登录一个真实账号 | 成功，且**刷新后仍是登录态** |
| 3 | 打开 `/blog` | 文章列表有内容，**未分类文章也在** |
| 4 | 看博客详情页 | 正文渲染、代码高亮正常 |
| 5 | 点赞 / 评论 | 成功，刷新后还在 |
| 6 | 签到 | 翻牌拿到鱼干，余额增加 |
| 7 | 再签一次 | 提示「今天已签到」 |
| 8 | 投喂一篇文章 | 自己扣、作者收 80% |
| 9 | `npm run cli -- fish balance <你自己>` | 与页面显示一致 |
| 10 | 管理员登录 → /admin | 能进；普通账号访问应被挡 |
| 11 | **传一张 >1MB 的图到图床** | 成功。若 413，是 nginx 的 `client_max_body_size`（见 §3.8），不是应用的问题 |

排障提示：

- **这 11 条可以一条命令跑掉**（凌晨挨个点最容易漏，而漏掉的往往就是坏的那条）：

```bash
cd web-next
npm run smoke -- --url https://raricy.com --user <核心用户名> --pass <密码>
```

它是**只读**的：不签到、不点赞、不发文；图床那条用「2MB 但内容非法」的文件探 ——
穿过 nginx 后被应用以 400 拒掉（校验在写盘前，也不消耗配额），一个字节都不会存。
唯一副作用是 `last_login` 被更新。实测跑完前后图床/签到/点赞/鱼干四项数据零变化。

它覆盖：HTTPS、公开页面、CSRF（外站拒 + 同源放行）、**登录态是否真的粘住**、
文章列表与详情、图床体积（nginx 413）、角色门控与登录回跳。

> 「登录态是否粘住」那条会照浏览器的规矩丢掉 http 站点下发的 Secure cookie ——
> 否则它会在自己最该抓的那个 bug 上给假绿（实测确认过）。

不给 `--user/--pass` 也能跑，只是跳过需要登录的部分。

排障提示：

- 第 3 步 **403** 且你确实登录了 → 该账号角色不够。`/blog` 需要 core 及以上
  （与 Flask 的 `@authenticated_required` 一致）。用
  `npm run cli -- promote-core <用户名>` 提权。
- 第 6/8 步 **503** → 账户服务没通，查 §3.5 那三个 `ACCOUNT_*` 变量。
  这是 fail-closed 的正常表现，不是崩溃。

## 4. 回滚

### 路线 A 的回滚

```bash
sudo systemctl stop raricy-next
# nginx 切回 Flask 的 upstream
sudo systemctl start raricy
```

⚠️ **库要不要恢复备份，取决于 Next 已经跑了多久**：
- 只跑了几分钟、没什么新数据 → 直接把 nginx 切回 Flask，但 Flask 读不了规整后的库，
  所以必须**同时**把 `DATABASE_URL` 指回原库（原库没被改过），期间 Next 写入的数据会丢。
- 已经跑了一段时间、有真实新数据 → 别恢复备份，改用路线 B 的兼容层让 Flask 读规整后的库：
  Flask 侧设 `DB_DATETIME_MS=true` 再启动。

### 路线 B 的回滚

nginx upstream 切回 Flask 即可，库是共用的，不丢数据。前提是 Flask 已经开着
`DB_DATETIME_MS=true`。

## 5. 彩排结论（在生产数据副本上实跑的结果）

整套流程在真实库副本上走过一遍（**源库全程只读**，事后校验和比对确认未被改动）。
下面这些数字是实测的，可以拿来当上线时的预期值：

| 项目 | 实测结果 |
|------|----------|
| 规整耗时 | **1.2 秒**，31 万个时间戳（311057 个）→ INTEGER 毫秒 |
| 行数损失 | 0（users 465、blogs 6183、blog_comments 63209 前后一致） |
| 墙上时间漂移 | **0** —— 33 个时间列 / 311057 个时间戳全量逐条比对，无一漂移 |
| 待补偿的未翻牌签到 | 7 条，补发 19 条小鱼干；重复执行不重复发放（幂等成立） |
| Next 冷启动 | 2 秒 |
| 首页 / 文章列表 | 200，列表 ~20ms |
| 账目一致性 | 签到 / CLI 充值 / 投喂三条路径，本地余额 == 远端余额 |
| fail-closed | 停掉账户服务后投喂 → **503，本地余额/流水/投喂记录三项零变化** |

彩排中确认的几个「不是 bug」的行为，上线时别误判：

- **首页不列文章** —— 它本来就是导航页，文章在 `/blog`。
- **匿名访问 `/blog` 返回 403** —— Flask 侧也是 `@authenticated_required`，行为一致。
- **新注册用户（role=user）访问 `/blog` 仍 403** —— 需要 core 及以上，与 Flask 一致。
- **没配 `ACCOUNT_SERVICE_INTERNAL_TOKEN` 时注册直接 503** —— fail-closed 的设计如此，
  不是故障。配上就好。

### `migrations/`（Alembic）—— 删之前先看这段

**现状**：schema 由 Flask-Migrate/Alembic 管（31 个版本 + `alembic_version` 表）。
而 `migrations/env.py` 第 4 行就是 `from flask import current_app` —— **Alembic 离了
Flask 根本跑不起来**。所以「删 Flask」= 顺手把 schema 的演进能力也删了，而文档另一处
又写着「绝不对该库跑 `prisma migrate`」。两句话合起来就是：**以后没人能改表结构。**

**出路**：`schema.prisma` 已经补成忠实反映生产库了 —— 22 张表、全部列、69 个索引
（含 3 个唯一索引的生产原名）一个不差，可以用它重建 schema：

```bash
cd web-next
npx prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script
```

**但别急着让 `prisma migrate` 接管这个库。** 我实测过，代价比看上去大：

| 实测现象 | 说明 |
|---|---|
| `prisma migrate dev` 报 **Drift detected** | 库是 SQLAlchemy 建的，列类型写作 `VARCHAR(36)`；Prisma 生成 `TEXT`。SQLite 里两者等价（类型亲和），但 Prisma 的 diff 认为不同 → **永久性 drift** |
| drift 之下 `migrate dev` 会**提议 reset 数据库** | 按下去就是清空生产库。**生产上永远别跑 `migrate dev`** |
| 一个「什么都没改」的 diff，生成的却是 `DROP TABLE "users"` … | Prisma 在 SQLite 上靠 RedefineTables 改表：建新表 → 拷数据 → 删旧表 → 改名。数据确实活下来（实测 465 用户完好），但它会把 155MB 库里**每一张表整个重写一遍** |
| 重写后多出 6 个重复的唯一索引 | `blog_feeds_blog_id_user_id_key` 等 —— 真实库把这些约束建成了表级 UNIQUE（匿名 `sqlite_autoindex_*`），Prisma 另建了具名索引。功能不冲突，但冗余 |

**所以推荐这样**：schema 变更本来就很少，别为它引入一次全库重写。

1. 手写 `ALTER TABLE` SQL，直接对库执行（**先备份**）；
2. `npx prisma db pull` 把 `schema.prisma` 同步成库的样子；
3. `npx prisma generate`；
4. 跑一遍 `npm test` + `npm run e2e` 确认没坏。

真要让 Prisma 管迁移的话，验证过的路子是 **`migrate diff` 生成 + `migrate deploy` 应用**
（`deploy` 不做 drift 检测，是生产该用的命令），并且 baseline 目录里
**必须有 `migration_lock.toml`**，否则报 `Could not determine the connector`：

```bash
mkdir -p prisma/migrations/0_init
printf 'provider = "sqlite"\n' > prisma/migrations/migration_lock.toml    # ← 少了它就报错
npx prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script \
  > prisma/migrations/0_init/migration.sql
npx prisma migrate resolve --applied 0_init      # 告诉 Prisma「这些已经在库里了」
```

之后每次改 schema：`migrate diff --from-schema-datasource --to-schema-datamodel` 生成
增量 → **先读一遍生成的 SQL**（它可能包含全表重写）→ `migrate deploy` 应用。

**baseline 之前别删 `migrations/`** —— 万一要回退，那 31 个版本是唯一的历史。

      - `web-next/scripts/check-permissions.mjs` + `package.json` 里的 `check:perms`
        —— 它靠读 `app/` 下的装饰器来比对权限档位，Flask 没了就无从比对。
        （脚本自己会在找不到 `app/` 时报错退出，不会假装通过；但留着一个必然
        失败的检查没意义。）权限仍有 e2e 把关：`tests/e2e/access-control.spec.ts`
