# 认证与会话迁移

迁移里最怕“用户全部要重置密码”。**好消息：不用。** 已实测双向互通。

## 1. 密码哈希：与 werkzeug 完全兼容（已实测）

现库 `password_hash` 是 werkzeug 生成的，现代默认是 **scrypt**：

```
scrypt:32768:8:1$<salt>$<hexhash>     # N=32768, r=8, p=1, dklen=64
```

也可能有历史的 `pbkdf2:sha256:<iter>$<salt>$<hash>`。

`src/lib/password.ts` 用 Node 内置 `crypto` 同时支持两者：

- `verifyPassword(plain, stored)`：按前缀分派到 scrypt / pbkdf2，恒定时间比较。
- `hashPassword(plain)`：生成 **werkzeug 可读** 的 scrypt 串（新注册/改密码用）。

### 实测结论（双向）

| 场景 | 结果 |
|---|---|
| werkzeug 生成的哈希 → Node `verifyPassword` | ✅ 正确密码 true / 错误密码 false |
| Node `hashPassword` 生成 → Python `check_password_hash` | ✅ True / False |

关键参数：scrypt 的 `maxmem` 要设到 ~132MB（`132*1024*1024`），否则 Node 默认 32MB 会
因内存上限抛错——`password.ts` 已按 werkzeug 的取值配好。

→ **用户无需改密码**，Flask 与 Next 可同时校验同一批哈希。

## 2. 会话：JWT 替代 Flask-Login

### 机制对齐

Flask-Login 用 itsdangerous 签名的 session cookie，存 `_user_id` / `session_version` / `_fresh`。
本方案换成 **JWT**（`jose` 库，HS256，用 `SECRET_KEY` 签名），载荷 `{ uid, sv }`：

- `sv` = 登录时的 `user.session_version` 快照。
- 每次 `getCurrentUser()`：验签 → 按 `uid` 载入用户 → **比对 `sv` 与 DB 里的 `sessionVersion`**，
  不一致即视为登出。**完全复刻**改密码/强制下线时“自增 session_version 让旧会话失效”的语义。

### 一个必须知道的代价

itsdangerous 的 Flask session 与 JWT **格式不互通**。所以切到 Next 时，**已登录用户会被登出一次，
需重新登录**（密码不用改）。这是可接受的一次性成本，务必在上线公告里说明。

若想连“重新登录”都省掉，得在 Next 里解析 Flask 的 session cookie（重写 itsdangerous 的
签名/序列化逻辑）——不推荐，收益不抵复杂度。

### Cookie 选项

`src/lib/session.ts`：`httpOnly` + `sameSite=lax` + 生产环境 `secure` + 30 天有效期。
`SECRET_KEY` **必须与 Flask 一致**（否则加密字段 `fish_api_key_encrypted` 等也会对不上）。

## 3. 角色与禁言

`src/lib/auth.ts` 复刻 `User` 模型的属性方法：

| Flask | Next |
|---|---|
| `is_owner` | `isOwner(user)` |
| `has_admin_rights` | `hasAdminRights(user)` |
| `is_core_user` | `isCoreUser(user)` |
| `is_currently_banned()` | `isCurrentlyBanned(user)`（含自动过期判定）|

装饰器 `@authenticated_required` 等在 Next 里的等价物：在 route/page 开头
`const user = await getCurrentUser(); if (!isCoreUser(user)) return apiErr(403, …)`。

> 注意：Flask 的 `admin_required` 失败返回 **HTML 403**，不适合 JSON API——Next 里统一
> 返回 `{ code: 403 }` JSON，天然规避了这个历史坑。

## 4. 敏感字段加密（fish_api_key_encrypted）

现用 Python `cryptography.Fernet`（`FISH_ENCRYPTION_KEY`，回退 `SECRET_KEY`）。
Next 侧对接账户微服务时需实现 **Fernet 兼容**的解密（Fernet = AES-128-CBC + HMAC-SHA256，
Base64 URL）。可用 `fernet`（npm）或手写。属账户模块迁移的一部分，见 [06](06-剩余工作与里程碑.md)。
