# 快速开始：把 web-next 跑起来

## 前置

- Node ≥ 20（本机实测 22.23）
- 现有 `instance/database/db.db` 存在

## 步骤

```bash
cd web-next

# 1. 安装依赖（已装则跳过）
npm install

# 2. 生成规整后的开发数据库副本（把线上 SQLite 复制成 prisma/dev.db 并转 ISO 时间戳）
npm run db:normalize

# 3. 配置环境变量
cp .env.example .env
#   .env 里：
#   DATABASE_URL="file:./dev.db"              ← 开发用副本；上线改指线上库
#   SECRET_KEY="<与 Flask .env 相同>"          ← 会话签名 + 加密字段一致性
#   AVATARS_DIR / IMAGE_UPLOAD_FOLDER          ← 指向真实 data/avatars、data/images
#   （头像/图床已由 Next 原生分发，不再需要 FLASK_ORIGIN 回源）

# 4. 生成 Prisma Client
npm run prisma:generate

# 5. 开发模式
npm run dev            # http://localhost:3000

# 或生产构建 + 启动
npm run build && npm start
```

## 部署在 nginx 反向代理后面（重要）

Next 的 CSRF 中间件用「浏览器 Origin」对比「本站对外 Host」。反代不透传原始 Host 时，
Next 收到的 `Host` 是上游地址（如 `127.0.0.1:3000`），与 Origin(`https://raricy.com`) 不符，
**正常的登录/写请求会被误判为跨源 → 403 `跨源请求被拒绝 (CSRF)`**。

中间件判定对外 Host 的顺序：`ALLOWED_ORIGINS` → `X-Forwarded-Host` → `Host`。
两种解法（任选其一，建议都做）：

### 解法 1：nginx 正确透传（推荐）

```nginx
location / {
    proxy_pass http://127.0.0.1:3000;
    proxy_http_version 1.1;

    # ↓ 关键：透传原始 Host / 协议，Next 才能算出正确的对外来源
    proxy_set_header Host              $host;
    proxy_set_header X-Forwarded-Host  $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header X-Real-IP         $remote_addr;
    proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;

    # WebSocket / SSE（如用到）
    proxy_set_header Upgrade    $http_upgrade;
    proxy_set_header Connection "upgrade";
}
```

### 解法 2：显式配置可信来源（反代配置不确定时最稳妥）

`.env.production`：
```bash
ALLOWED_ORIGINS="raricy.com"      # 多个用逗号分隔；可写 https://raricy.com
```

### 自检

```bash
# 模拟反代场景：不应返回 403（返回 401 说明已通过 CSRF 进到登录逻辑）
curl -s -o /dev/null -w "%{http_code}\n" -X POST https://你的域名/api/auth/login \
  -H "Origin: https://你的域名" -H "Content-Type: application/json" \
  -d '{"username":"__nobody__","password":"x"}'
```

> 注：`X-Forwarded-Host` 仅在由反代覆写时可信（nginx 应配 `proxy_set_header X-Forwarded-Host $host`）。
> 浏览器跨站请求无法附加该自定义头（会触发 CORS 预检且本站不放行），不构成绕过面；
> 但若把 Next 裸暴露公网，请务必配 `ALLOWED_ORIGINS` 作为权威来源。

## 部署自检脚本（先跑这个）

遇到登录 / CSRF / 会话问题，先跑自检，它会直接指出原因，不用逐个猜：

```bash
cd web-next
npm run diagnose                              # 只查本地配置与数据库
npm run diagnose -- --url https://你的域名      # 额外做线上活体检查
```

覆盖：`SECRET_KEY` 缺失、`DATABASE_URL` 路径/权限/WAL 目录可写、
**时间戳是否已规整**（用 `CAST AS TEXT` 取回真实格式 + 实际让 Prisma 反序列化整行 users）、
`ALLOWED_ORIGINS`、HTTP 下的 Secure cookie 问题、CSRF 是否仍在拦截、登录接口实际状态码。

## 症状 → 原因速查

### A. 登录显示「登录成功」，但刷新仍未登录（无任何报错）

**会话 cookie 被浏览器丢弃**。站点走纯 HTTP 时，带 `Secure` 标记的 cookie 会被直接丢掉。

判定顺序：`COOKIE_SECURE`（显式）→ `X-Forwarded-Proto`（反代透传的真实协议）→ `NODE_ENV`。
- nginx 已配 `proxy_set_header X-Forwarded-Proto $scheme` → 自动正确，无需额外配置。
- 否则在 `.env.production` 显式设 `COOKIE_SECURE="false"`。
- 验证：`curl -sD- -o/dev/null -X POST .../api/auth/login -d '...'` 看 `Set-Cookie` 是否含 `Secure`。

> ⚠️ HTTP 下会话 cookie 明文传输，链路上任何人可窃取并冒用会话。
> `COOKIE_SECURE=false` 只是过渡期的显式选择，生产站点请上 TLS。
> 原 Flask 未设 `SESSION_COOKIE_SECURE`（默认 False），故老站在 HTTP 下一直可用。

### B. 正常请求被拒：`403 跨源请求被拒绝 (CSRF)`

反代未透传原始 Host，见上文「部署在 nginx 反向代理后面」。

### D. 启动即 `TypeError: Cannot read properties of undefined (reading 'map')`

**Next 版本装错了**（跨了大版本）。本项目对着 **Next 15** 写（`package.json` 锁 `^15.1.4`，
`package-lock.json` 锁 `15.5.20`）。实际发生过：线上跑成 **Next 16.2.10**，启动即抛该错，
且堆栈指向 `ignore-listed frames`，完全看不出根因。

```bash
npx next --version        # 应为 15.x
npm run diagnose          # 「0. 运行时版本」会直接点名不匹配
```

**装依赖一律用 `npm ci`**（严格按 lockfile 安装），不要用 `npm install next@latest` /
`npm update` —— 那会绕过 lockfile 把 next 升到 16。若已经装错：

```bash
cd web-next
rm -rf node_modules
npm ci                    # 严格复原 lockfile 锁定的 15.5.20
npm run build && <重启服务>
```

### C. 登录返回 500

1. **时间戳未规整（最常见）**：库里若仍是 SQLAlchemy 的空格格式（`2025-08-09 20:48:45.776483`），
   Prisma 在**驱动层**解析 DATETIME 列即抛 `Conversion failed: input contains invalid characters`（P2023）。
   登录必撞——`getCurrentUser()` 的 `findUnique` select 了 `createdAt`/`lastLogin`/`banUntil`。
   → 对**目标库**跑 `node scripts/normalize-datetimes.mjs`。
   **⚠️ 规整不可逆，且规整后 Flask 读不了该库**——若新旧站共用同一个库，务必先读
   [03 §3「必须知道的取舍」](03-数据库映射与陷阱.md)（硬切换 / 用附录 A 的兼容层灰度 / 用副本验证）。
2. **`SECRET_KEY` 未配置**：`src/lib/session.ts` 会直接 `throw new Error('SECRET_KEY 未配置')` → 500。
3. **数据库路径/权限**：`DATABASE_URL` 指向的 SQLite 文件是否存在、进程是否可读写（WAL 需要写目录权限）。
4. 看真实堆栈：`pm2 logs` / `journalctl -u <service>` / 前台 `npm start` 直接看输出。
