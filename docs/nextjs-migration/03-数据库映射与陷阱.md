# 数据库迁移文档（核心）

本篇是你特别要求的“详细数据库迁移文档”。它回答三件事：
**数据怎么过去、schema 怎么映射、有哪些坑（以及已实测的解决方案）**。

---

## 1. 迁移策略：不迁数据，直接复用同一个库

现有库是 `instance/database/db.db`（SQLite）。Prisma 用 `provider = "sqlite"` **直连它**：

```
# web-next/.env
DATABASE_URL="file:../../instance/database/db.db"   # 相对 prisma/ 目录
```

- **不导出/导入数据**，不换数据库引擎。
- Next 与 Flask 读写**同一张表**，字段、主键、约束原样不动。
- 未来若要上 Postgres，是另一件事（见 §7），本次不做——SQLite 直连是风险最低的切换点。

> ⚠️ 唯一的现实前提：**时间戳格式**必须先规整（见 §3），否则 Prisma 读不了。这是本迁移
> 最关键、也最容易被忽略的坑，已在真实库上复现并给出脚本。

---

## 2. Schema 映射总览

`web-next/prisma/schema.prisma` 已从真实 DDL 1:1 映射，共 **20 个模型**：

| 领域 | 模型（表名） |
|---|---|
| 用户/权限 | `User(users)`、`InviteCode(invite_codes)`、`UserBan(user_bans)` |
| 博客 | `Blog(blogs)`、`BlogContent(blog_contents)`、`BlogLike(blog_likes)`、`BlogFeed(blog_feeds)` |
| 分类/评论 | `Category(categories)`、`BlogComment(blog_comments)`、`CommentLike(comment_likes)` |
| 互动/内容 | `Notification(notifications)`、`ClipBoard/ClipText`、`ImageHosting`、`PhotoWallItem`、`Vote/VoteOption/VoteRecord`、`DailyCheckIn` |
| 经济/审计 | `FishTransaction`、`AdminActionLog`、`AdminActionAppeal` |

映射约定：

- **字段名** camelCase，用 `@map("snake_case")` 对齐真实列；**表名** 用 `@@map`。→ 代码里写 `blog.createdAt`，库里仍是 `created_at`，**不改结构**。
- **主键**：UUID/短 ID 表用 `String @id`；自增表用 `Int @id @default(autoincrement())`。与现状一致。
- **唯一约束**沿用原名：`uq_blog_like_blog_user`、`uq_vote_user`、`uq_user_checkin_date` 等。
- **自关联**：`Category` 的父子（`CategoryTree`）、`BlogComment` 的 `parent/root`（`CommentParent`/`CommentRoot`）都用命名关系显式声明。
- **多重外键到 User** 用命名关系区分（如通知的 `recipient`/`actor`、审计的 `admin`/`targetUser`）。

## 3. 头号陷阱：DateTime 文本格式不兼容（已实测）

### 现象

SQLAlchemy 把时间写成 **Python 风格**：

```
2026-07-12 06:09:17.817454      ← 空格分隔、6 位微秒、无时区
2026-07-15                      ← 纯日期（checkin_date）
```

Prisma 的 SQLite 连接器**按列声明类型 `DATETIME` 在驱动层解析**，要求 ISO-8601。
直接读会抛：

```
Inconsistent column data: Conversion failed: input contains invalid characters
```

> 已实测复现：`prisma.user.findFirst()` 直接报此错。**注意**：把 Prisma 字段声明成 `String`
> **也没用**——驱动层是按“列的声明类型”解析的，不是按 schema 字段类型。

### 解决：一次性把时间戳规整为 INTEGER（Unix 毫秒）

脚本 `web-next/scripts/normalize-datetimes.mjs`（用 Node 内置 `node:sqlite`，无新依赖）：

```bash
cd web-next
npm run db:normalize
# 等价：把线上库复制成 prisma/dev.db，再把所有 DATETIME/DATE 列转成
#       INTEGER 毫秒（幂等：只转 typeof='text' 的值）
```

覆盖全部 22 处时间列（脚本里已枚举）。实测对真实库 **311,057 个时间戳零丢失**。

#### 🔴 为什么是 INTEGER 而不是 ISO 文本（2026-07-16 补测试时发现的重大问题）

本脚本**曾经**把时间戳转成 ISO 文本 `2025-08-09T20:48:45.776Z`。Prisma 能正常**读**，
所以一路看着都没问题 —— 但这是个陷阱：

**Prisma 做 DateTime 比较时绑定的是 INTEGER 毫秒**，而 SQLite 跨存储类型比较
**按类型序**（`NULL < INTEGER < TEXT < BLOB`）而非数值。于是对 TEXT 存储的行：

| 查询 | 实际行为 |
|---|---|
| `createdAt: { gte: X }` | **所有 TEXT 行恒真**（TEXT > INTEGER） |
| `createdAt: { lt: X }` / `lte` | **所有 TEXT 行恒假** |

**实测后果**：`countBlogsToday()` 用 `createdAt: { gte: 今天零点 }` 统计当日发文数 ——
TEXT 存储下它把用户**历史上全部文章**都算成「今天发的」。真实库里发文最多的用户有
**524 篇**（前三名 524/510/426）→ 切换后立刻永久触发「今日发布已达上限（20篇）」，
**再也发不了文**。而且这类错误静默无报错，只会表现为「功能莫名其妙不能用」。

存成 INTEGER 后，读 / 写 / 比较 / 排序全部一致且正确（已用真实库实测）。
回归用例见 `tests/service/blog-service.test.ts` 的「回归：日期比较必须按数值而非存储类型」。

#### 连带影响：裸 SQL 的日期函数不可用

存 INTEGER 后，SQLite 的 `date()` / `strftime()` **无法解析**（它们会把整数当儒略日）。
因此**不要在 `$queryRaw` 里对时间列用日期函数**，改用 Prisma 原生范围查询。
`getTodayCheckinFish` 就踩过这个坑（曾用 `date(created_at) = ?` → 恒不匹配 → 今日签到永远显示 0）。

#### 自检

```bash
npm run diagnose        # 会直接报出「空格未规整 / TEXT 存储 / 混存」三种问题
```

### ⚠️ 这里有个必须知道的取舍

规整成 ISO 后，**SQLAlchemy 反而读不了**（它的 SQLite DATETIME 期望空格/微秒那套正则）。
也就是说同一列**不能**让两套 ORM 都满意。三条路：

1. **硬切换（推荐用于最终上线）**：停 Flask → 规整线上库 → 起 Next。简单、干净、一次到位。
2. **灰度共存**：给 Flask 侧的 SQLAlchemy 配一个 **ISO 兼容 DATETIME 类型**（`TypeDecorator`，
   见 §附录 A），让 Flask 也按 ISO 读写。改动集中、可控，但要动 Flask。
3. **本地验证/开发**：就用 `npm run db:normalize` 生成的 `prisma/dev.db` 副本，**不碰线上库**，
   Next 单独针对副本开发（本次地基就是这么跑通的）。

> 本仓库当前 `web-next/.env` 指向 `file:./dev.db`（规整后的副本），正式上线时改成线上库路径
> 并选 §1/§2 的路线。

## 4. 其它类型陷阱

| 列 | 现状 | Prisma 处理 |
|---|---|---|
| `admin_action_logs.extra` | SQLite 列**声明类型 `JSON`** | ⚠️ 已实测：Prisma 的 SQLite 连接器**按声明类型读取,即使 schema 映射成 `String` 也会报 `Value JSON not supported`**（与 DateTime 同类根因）。解决:typed 查询里**不要 select `extra`**,改用 `$queryRawUnsafe("SELECT id, CAST(extra AS TEXT) AS extra …")` 单独取,再 `JSON.parse` guarded。见 `src/lib/audit-service.ts` 的 `listPublicLogs`。此法对线上库同样适用。|
| 各 `Boolean` | 存 0/1 | Prisma 原生兼容，无需处理 |
| `dried_fish` / 坐标 / `amount` | `Float` | 原样 `Float` |
| 短 ID / UUID 主键 | `String` | 原样，**不要**让 Prisma 生成默认值（沿用应用层生成逻辑）|

## 5. 写入时的时间戳一致性

切换后由 Next 写入的新行，Prisma 会写 ISO 格式（`...T...Z`）。若仍处于“Flask 也在写”的
灰度期，会出现同表**混合两种格式** → 必须走 §3 路线 2（让 Flask 也用 ISO），否则读方会各自炸。
**这也是为什么推荐硬切换**：迁移期尽量短，避免双写不同格式。

## 6. 迁移执行清单（正式上线时）

```text
[ ] 0. 全量备份 instance/database/db.db（cp 一份带时间戳）
[ ] 1. 冻结写入：停 Flask（或置只读维护页）
[ ] 2. 运行规整：node scripts/normalize-datetimes.mjs --in-place <线上库路径>
       （先在备份副本上演练一遍，确认 21+ 行更新、无报错）
[ ] 3. 配置 web-next/.env：DATABASE_URL 指向线上库；SECRET_KEY 与 Flask 一致；
       FISH_ENCRYPTION_KEY / ACCOUNT_* 照搬
[ ] 4. npm run prisma:generate && npm run build
[ ] 5. 起 Next（PM2/systemd），Nginx 把流量切到 3000
[ ] 6. 冒烟：登录、列表、详情、点赞、退出
[ ] 7. 头像/图床：先靠 next.config rewrites 回源 Flask 静态目录，或迁移到 Next 的 /public + 路由
[ ] 8. 回滚预案：Nginx 切回 Flask + 用备份库（因已规整成 ISO，回滚需用备份的原始库）
```

> 第 8 点注意：一旦线上库规整成 ISO，Flask 就读不了了；**回滚必须依赖第 0 步的备份**，
> 或事先给 Flask 打好 §附录 A 的兼容类型。

## 7. 未来：迁到 Postgres（可选，非本次）

SQLite 直连是过渡最优解，但并发写、多实例部署终会受限。届时：

1. `schema.prisma` 的 `provider` 改 `postgresql`，`DATABASE_URL` 换 PG。
2. 用 `prisma migrate diff` 从当前 schema 生成建表 SQL。
3. 数据搬迁：`pgloader sqlite://… postgresql://…`，或写一次性 ETL（注意 Boolean/JSON/时间列）。
4. 此时时间列可直接用 `timestamptz`，§3 的坑自然消失。

## 附录 A：让 Flask 也读 ISO（灰度共存用）

在 Flask 侧定义一个兼容 DATETIME 类型，读写都用 ISO，从而与规整后的库、与 Prisma 一致：

```python
# app/extensions/iso_datetime.py
import datetime as dt
from sqlalchemy.types import TypeDecorator, String

class ISODateTime(TypeDecorator):
    """把 datetime 以 ISO-8601 (…T…Z) 存取，兼容 Prisma 规整后的库。"""
    impl = String
    cache_ok = True
    def process_bind_param(self, value, dialect):
        if value is None: return None
        if value.tzinfo is None:
            value = value.replace(tzinfo=dt.timezone.utc)
        return value.astimezone(dt.timezone.utc).strftime('%Y-%m-%dT%H:%M:%S.') \
               + f'{int(value.microsecond/1000):03d}Z'
    def process_result_value(self, value, dialect):
        if value is None: return None
        return dt.datetime.strptime(value, '%Y-%m-%dT%H:%M:%S.%fZ')
```

把模型里的 `db.DateTime` 换成 `ISODateTime`（可用基类/mixin 批量替换）。这是灰度期
双栈共存的正解；若走硬切换则不需要。
